# Design — my-profile / iOS / my-profile

## Current context

Текущая iOS-оболочка уже имеет проверку авторизации и три вкладки. `RootView.swift`
показывает `ContentView()` только для активной сессии, а `ContentView.swift`
содержит вкладку профиля как `ContentUnavailableView`. `AuthFeature` является
существующим пакетом Swift с проверкой сессии, секретом в Keychain и публичным
методом `AuthSessionModel.invalidateSession`. Истории интервью и операции
выхода в текущем iOS-коде нет, поэтому новая фича должна добавить поведение
профиля без переноса его в оболочку приложения.

## Proposed architecture and boundaries

Создаётся новый пакет Swift `MyProfileFeature`. Он владеет профильной фичей,
машиной состояний, сценариями использования, DTO, пагинацией и SwiftUI-поверхностью.
Публичная поверхность ограничивается фабрикой или входной view и типизированными
зависимостями: базовый адрес API, возможность запроса в рамках сессии или
поставщик токена сессии, callback восстановления после выхода и абстракции часов
для тестов. `SysDevScen` только подключает пакет, создаёт зависимости и вставляет
профильную view во вкладку профиля.

## Data and control flow

Появление вкладки профиля запускает принадлежащую экрану задачу загрузки.
Сценарий использования запрашивает текущий email профиля, затем читает
`GET /api/interviews/history` страницами до `page.hasMore = false`. ViewModel
публикует состояния загрузки, содержимого, пустой истории, ошибки истории или
сети, выполнения выхода и ошибки выхода. Нажатие «Мои интервью» при известном
количестве больше нуля создаёт событие краткого сообщения. Нажатие «Выход»
вызывает `POST /api/auth/logout`, затем через стык авторизации переводит оболочку
в состояние без сессии. Ответ 401 из профиля или истории вызывает тот же путь
восстановления.

## Dependency injection

`SysDevScenApp` остаётся корнем композиции: он уже создаёт конфигурацию
авторизации и может создать `MyProfileFeatureFactory`. Пакет авторизации должен
предоставить минимальный типизированный стык для запросов в рамках сессии или
доступа к токену без раскрытия реализации Keychain. Тесты получают заглушку
репозитория или клиента и callback выхода, чтобы не зависеть от живого API.

## Error and recovery model

Ошибка истории не блокирует «Выход» и оставляет «Мои интервью» недоступной до
успешной повторной загрузки. Offline-маппинг показывает восстановимое состояние
с retry. Сетевой сбой выхода оставляет пользователя на вкладке профиля и не
очищает UI так, будто выход успешен. Недействительная сессия очищает отображаемые
профильные данные и вызывает восстановление авторизации. Backend trace id не
выводится в основной UI.

## Concurrency model

ViewModel изолирован на `MainActor`. Задача загрузки принадлежит ViewModel и
отменяется при уходе владельца. Повторная загрузка использует политику отмены
и замены либо единственного активного запроса, чтобы не получить устаревший
счётчик. Пагинация выполняется последовательно и проверяет отмену между
страницами. Сетевой клиент остаётся асинхронным и не хранит изменяемое общее
состояние вне actor или value state.

## Security and data handling

Фича отображает только email и счётчик. Детали истории, оценка, название кейса
и идентификаторы сессии не показываются. Секрет сессии не логируется, не
сохраняется в состоянии профиля и не выходит за типизированный стык. Политика
только HTTPS сохраняется. Выход и недействительная сессия очищают отображаемые
персональные данные до возврата в поток авторизации.

## Platform SDK considerations

Доказательства проекта показывают `IPHONEOS_DEPLOYMENT_TARGET = 26.0` для
целевого приложения и `SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor`; доказательства
пакета показывают `swift-tools-version: 6.3` и нижнюю границу iOS 18.0 для
существующего пакета. Новый пакет должен обнаружить и согласовать собственную
платформенную границу на этапе Plan. Доступность Liquid Glass не предполагается:
`platform-ux.md` задаёт fallback для older-OS/SDK и настроек доступности.

## Design-system and accessibility

`platform-ux.md` является источником нативного UX для сопоставления компонентов.
SwiftUI-view использует системные standard components, профильный маркер
SF Symbols, семантические роли semantic roles, Dynamic Type, признаки VoiceOver
и нецветовые признаки состояния non-color state cues. Фича не вводит локальные
цветовые литералы вместо semantic roles и не создаёт custom glass effects без
доказанного разрыва.

## Migration and rollout

Реализация может добавляться за композиционным стыком: сначала пакет и тесты,
затем подключение пакета в проект Xcode, затем замена заглушки вкладки профиля.
Откат ограничен отключением композиции пакета и возвратом заглушки. Общее
продуктовое поведение не меняется.

## Alternatives and trade-offs

Размещение поведения профиля прямо в целевом приложении отклонено: фича включает
UI, доменное состояние, сеть, пагинацию и восстановление, что нарушило бы
allowlist оболочки. Расширять `AuthFeature` всей профильной логикой отклонено:
пакет авторизации должен остаться владельцем авторизации и сессии, а не поведения
профиля и истории. Новый пакет даёт явную границу, но требует подключения Xcode
и минимального стыка сессии.

## Modularity decision

- Outcome: isolated
- Capability triggers: independent-feature=yes; domain-data=yes; network=yes; persistence=no; reusable-ui=no; consumers=1; independent-ownership=yes
- Physical boundaries: proposed Swift package MyProfileFeature capability unit
- Public contracts and dependency direction: MyProfileFeature открывает минимальную фабрику и типизированный стык API сессии; SysDevScen потребляет публичный контракт; MyProfileFeature зависит от контрактов и скрывает детали реализации
- App-shell responsibilities: entry-points, lifecycle, root-routing, dependency-wiring, platform-configuration, target-resources
- App-shell capability ownership: none
- Repository evidence: iOS/AuthFeature/Package.swift; iOS/SysDevScen/SysDevScen.xcodeproj/project.pbxproj; iOS/SysDevScen/SysDevScen/ContentView.swift; iOS/SysDevScen/SysDevScen/RootView.swift
- Rationale and trade-offs: Отдельная feature имеет network и domain state, поэтому proposed Swift package даёт проверяемую границу, типизированную публичную поверхность и focused tests без расширения композиционного слоя.
- Migration boundary and trigger: Типизированная фабрика и стык сессии позволяют подключить пакет сейчас; триггер миграции появится при втором потребителе или альтернативном поставщике сессии, требующем развития контракта.
- Over-modularization check: Гранулярность соответствует связной профильной возможности, а не папке или слою; пакет объединяет UI, состояние, сценарии использования и API-стык одного поведения.
- Boundary guard verdict: PASS

## Applied engineering scopes

- application: Интеграция вкладки профиля и восстановление авторизации затрагивают корневую композицию, поэтому дизайн ограничивает изменения приложения сборкой зависимостей и заменой содержимого вкладки.
- concurrency: Асинхронная загрузка профиля, пагинация и выход требуют состояния на `MainActor`, принадлежащих задач, отмены и защиты от устаревших обновлений.
- localization: Русские пользовательские строки требуют механизма локализации проекта или явного решения по ресурсу без конкатенации фраз.
- networking: Запросы профиля, постраничной истории и выхода требуют HTTPS, передачи cookie или сессии, пагинации и маппинга ошибок.
- package: Отдельный пакет Swift нужен как физическая граница связной профильной возможности и проверяемая единица владения.
- ui: SwiftUI-поверхность, адаптация Liquid Glass, доступность, Dynamic Type и сценарии симулятора являются нативными UI-обязательствами.

## Platform UX trace and decisions

`platform-ux.md` фиксирует нативное iOS-сопоставление и переводит shared intent
в конкретные решения для экрана профиля. Используется стандартный профильный
символ `standard profile symbol`, `functional controls` и `navigation` только
для `Liquid Glass`, спокойный `content background`, `soft blue` через
`semantic roles`, `critical logout role`, `Dynamic Type`, `VoiceOver`
`announcements`, `Reduce Transparency`, `Reduce Motion` и `older-OS/SDK fallback`.
Архитектура должна удержать эти решения
внутри пакета фичи, открывая состояние и действия, которые нативные элементы
могут представить без владения custom effect в композиции.

## System-design checklist

- Поведение фичи, сеть и состояние изолированы в `MyProfileFeature`.
- Публичный API остаётся типизированным и минимальным.
- Направление зависимостей идёт от композиции `SysDevScen` к контракту пакета.
- Интеграция авторизации является стыком, а не общим изменяемым глобальным состоянием.
- Верификация имеет тесты пакета, сборку потребителя, UI-сценарии и проверки
  границ до архивирования реализации.

## Verification strategy

Verify объединит unit-тесты пакета для API, пагинации и маппинга состояния,
интеграционную сборку потребителя для подключения Xcode, UI-тесты симулятора
для поведения вкладки профиля, проверки доступности VoiceOver и Dynamic Type,
а также записи нативных UX-обязательств для внешнего вида, motion и fallback.
Нестабильность живого endpoint не создаёт детерминированный PASS; тесты должны
использовать заглушки или фикстуры для контрактных исходов и записывать живую
диагностику только как дополнительное доказательство.
