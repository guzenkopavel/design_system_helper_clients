## Goal

Реализовать сквозные тесты интерфейса c заглушкой по аргументу запуска,
покрывающие проверку сессии при запуске, ветвление шагов, возврат,
состояния ошибок, успешный вход и регистрацию, паритет текстов и шагов
c общим контрактом. Тесты детерминированы без обращения к живому серверу.

## Inline contract context

- `REQ-1` — приложение целиком за авторизацией, тесты подтверждают проверку при запуске.
- `REQ-2` — флоу начинается c ввода почты, тесты подтверждают ветвление и возврат.
- `REQ-3` — успех входа или регистрации немедленно открывает оболочку.
- `REQ-9` — наблюдаемое поведение, тексты шагов и правила совпадают c общим контрактом.
- `AC-1` — запуск без валидной сессии открывает экран входа.
- `AC-2` — почта существующего аккаунта ведёт к шагу с заголовком «Вход».
- `AC-5` — верные учётные данные открывают оболочку.
- `AC-25` — сквозной проход демонстрирует одинаковые шаги и тексты.
- `IOS-AC-4` — запуск без сессии: загрузка, затем шаг почты; оболочка недоступна.
- `IOS-AC-6` — ветвление входа или регистрации и возврат c сохранением почты.
- `IOS-AC-7` — успех — оболочка c кейсами; возврат невозможен.
- `IOS-AC-17` — паритетный проход шагов и текстов.

## Implementation deliverables

- Модификация тестовой цели c набором сквозных сценариев на заглушке, включаемой аргументом запуска в корневой композиции, без обращения к живому серверу и c детерминированными результатами каждого сценария.
- Сценарий проверки сессии при запуске, подтверждающий спокойное состояние загрузки и переход к шагу почты без отображения навигации оболочки, c проверкой идентификатора вспомогательных технологий корневой поверхности.
- Сценарий ветвления шагов, проверяющий заголовок входа для существующей почты и регистрации для новой почты, возврат к шагу почты c сохранённым значением поля и видимость почты на шаге пароля.
- Сценарий успешного входа и регистрации, подтверждающий немедленный переход в оболочку c активным разделом кейсов и отсутствие навигационного возврата в флоу.
- Паритетный проход, сверяющий видимые тексты шагов, действий и заголовков c общим контрактом для подтверждения одинакового наблюдаемого поведения обоих клиентов.

## Steps

1. Создать файл тестов интерфейса c настройкой c продолжением после сбоя и аргументом запуска заглушки.
2. Написать тест запуска без сессии: запуск, ожидание загрузки, проверка видимости поля почты и действия продолжения; проверка отсутствия навигации оболочки.
3. Написать тест ветвления входа: ввод существующей почты, продолжение, проверка заголовка входа и видимости почты.
4. Написать тест ветвления регистрации: ввод новой почты, продолжение, проверка заголовка регистрации.
5. Написать тест возврата к почте: ввод почты, переход к паролю, возврат, проверка сохранённого значения.
6. Написать тест успешного входа: ввод существующей почты, пароль, вход, проверка видимости кейсов и отсутствия возврата.
7. Написать тест успешной регистрации: ввод новой почты, пароль, регистрация, проверка кейсов.
8. Написать тест паритета: сквозной проход c проверкой всех текстов.
9. Запустить тесты c watchdog 300 секунд.
