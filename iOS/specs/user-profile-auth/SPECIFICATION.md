## Platform requirements

### IOS-REQ-1 — Root composition boundary

Сохранённое baseline-обязательство `app-shell`, расширенное гейтом запуска.
`SysDevScenApp` и корневая SwiftUI-композиция владеют только входом,
жизненным циклом, корневой маршрутизацией (теперь включая маршрутизацию
«флоу авторизации либо оболочка» по состоянию сессии), связыванием
зависимостей, платформенной настройкой и ресурсами target. Application target
не получает отдельную feature/data/network/persistence/reusable-UI
capability: auth-возможность подключается как внешняя единица, а не
реализуется внутри оболочки. Traces: `REQ-1`, `REQ-4`.

### IOS-REQ-2 — Native control semantics

Сохранённое baseline-обязательство, распространённое на поверхность
авторизации. Корневая оболочка и новый флоу используют стандартные
SwiftUI-средства навигации и управления. Русские подписи совпадают с
доступными именами; появление ошибок, загрузки и ограничения попыток
объявляется ассистивно; смена шага переводит ассистивный фокус к логическому
началу нового шага; состояния различимы не только цветом; поддержаны
`Dynamic Type`, светлое, тёмное и повышенно-контрастное оформление. Нативный
вид следует READY-решению `platform-ux.md`. Traces: `REQ-8`.

### IOS-REQ-3 — Template isolation

Сохранённое baseline-обязательство. Корневая поверхность не содержит прежний
Core Data список, действия изменения или UI-взаимодействия с хранением
шаблона. Реализация не добавляет аналитику, внешние действия, выход из
аккаунта, сброс пароля или иные границы вне утверждённого продуктового scope;
авторизация входит в утверждённый scope этой фичи. Traces: `REQ-1`.

### IOS-REQ-4 — Launch session gate

При запуске приложение определяет состояние серверной сессии, показывая
спокойное состояние загрузки без мигания формой входа. Валидная сессия ведёт
сразу в оболочку с активным разделом «Кейсы»; отсутствие или невалидность
сессии ведёт на шаг почты. До успешного входа никакая навигация оболочки не
отображается и недоступна. Повторные запуски в пределах жизни сессии (до 14
дней по контракту бэкенда) открывают оболочку без экрана входа.
Traces: `REQ-1`, `REQ-4`.

### IOS-REQ-5 — Email-first flow and branching

Флоу авторизации начинается с шага почты с одним основным действием
«Продолжить». По результату серверной проверки почты открывается шаг пароля с
заголовком «Вход» (аккаунт существует) либо «Регистрация» (почта новая);
введённая почта остаётся видимой на шаге пароля. Явное действие возврата (и
равнозначный навигационный возврат iOS) ведёт с шага пароля на шаг почты с
сохранённым и доступным для правки значением почты. Поле пароля скрывает
ввод; раскрытие видимости — явное управляемое действие. Traces: `REQ-2`.

### IOS-REQ-6 — Immediate shell entry on success

Верный пароль существующего аккаунта (вход) или валидная новая пара
почта/пароль (регистрация) немедленно открывает оболочку с активным разделом
«Кейсы». После успеха флоу закрывается целиком: вернуться в него жестом или
навигационным возвратом из оболочки нельзя. Traces: `REQ-3`.

### IOS-REQ-7 — Recoverable flow states

Все состояния флоу наблюдаемы, различимы и восстановимы:

- пустая почта или пустой пароль не отправляются; попытка продолжить
  показывает человекочитаемое объяснение рядом с полем, основное действие
  остаётся видимым и доступным;
- синтаксически невалидная почта и пароль короче 6 символов при регистрации
  отклоняются до сетевого запроса с понятным сообщением; серверная ошибка
  валидации (`422`) показывается рядом с формой в том же наблюдаемом стиле;
- неверные учётные данные (`401` на шаге входа) сообщаются без раскрытия
  существования аккаунта; поле пароля очищается, повтор доступен сразу;
- занятая почта при регистрации (`409`) объясняет, что аккаунт уже
  существует, и предлагает выполнить вход;
- ограничение попыток (`429` с `Retry-After`) — отдельное состояние с
  указанием времени повтора; до истечения срока основное действие не
  отправляет запрос; появление состояния объявляется ассистивно;
- отсутствие сети — отдельное восстановимое состояние с возможностью
  повторить; введённая почта сохраняется, пароль — нет.

Traces: `REQ-5`.

### IOS-REQ-8 — Session invalidation recovery

Истечение или отзыв сессии (ответ `401` на персональный запрос) возвращает
пользователя на шаг почты с коротким объяснением. Персональные данные прошлой
сессии перестают отображаться, а недействительный сессионный секрет очищается
из хранилища устройства до показа экрана входа. Traces: `REQ-6`.

### IOS-REQ-9 — Device secret protection

Обязательства хранения формулируются платформенно и наблюдаемо, без выбора
конкретного API за архитектуру:

- значение сессионного cookie хранится в защищённом системном хранилище
  класса `Keychain` так, что оно недоступно другим приложениям и не попадает
  в незащищённые файлы, `UserDefaults`-аналоги или логи;
- пароль не сохраняется на устройстве ни в каком виде, не логируется и не
  отображается после отправки;
- недействительный сессионный секрет удаляется из хранилища при возврате на
  экран входа;
- учётные и сессионные данные передаются только по защищённому каналу.

Traces: `REQ-7`.

### IOS-REQ-10 — Backend API client contract

Клиент к существующему API бэкенда покрывает четыре операции: проверку почты
(`email-check`), вход (`login`), регистрацию (`register`) и персональный
запрос (`profile`). Наблюдаемый контракт: запросы уходят только по `https`
(default `ATS` не ослабляется); единый формат ошибок бэкенда (код, сообщение,
идентификатор трассировки) отображается в состояния `IOS-REQ-7` без показа
технических кодов в основном тексте; серверная cookie-сессия принимается и
предъявляется в персональных запросах; ответ `401` персонального запроса
маршрутизируется в `IOS-REQ-8`. Адрес API — внешний параметр окружения.
Traces: `REQ-5`, `REQ-6`.

### IOS-REQ-11 — Async flow discipline

Проверка сессии, проверка почты и отправка входа/регистрации выполняются
асинхронно без блокировки интерфейса. Во время активного запроса основное
действие шага заблокировано с видимым индикатором, повторная активация не
порождает второй запрос. Уход с шага или возврат к правке почты отменяет
незавершённый запрос этого шага; отменённый запрос не меняет видимое
состояние задним числом. Traces: `REQ-5`.

### IOS-REQ-12 — Auth capability physical unit

По modularity v1 новая связная auth-возможность (флоу, клиент API, хранение
сессии) размещается в отдельной физической единице — `Swift package` с
собственными модульными тестами — за пределами application target. Application
target только компонует эту единицу через корневую композицию и не содержит её
реализацию. Точную структуру единицы фиксирует `design.md` с verdict boundary
guard. Traces: `REQ-1`.

### IOS-REQ-13 — Cross-client parity

Наблюдаемые шаги, тексты (включая «Продолжить», заголовки «Вход» и
«Регистрация», стартовый раздел «Кейсы»), порядок состояний и правила
гейтинга совпадают с общим контрактом и допускают только нативную
вариативность, явно разрешённую разделом `Allowed platform variance` файла
`specs/product/user-profile-auth/ux.md` (клавиатуры, контроль раскрытия
пароля, системный возврат, автозаполнение). Traces: `REQ-9`.

## Platform acceptance criteria

### IOS-AC-1 — Root composition and gate ownership

Статический review владения подтверждает: гейт запуска и маршрутизация
«флоу либо оболочка» живут в корневой композиции, application target не
содержит реализацию auth/network/storage capability, baseline-обязанности
оболочки не расширены.
Covers: IOS-REQ-1

### IOS-AC-2 — Native semantic interaction

Ассистивный сценарий на simulator показывает: имена полей и действий
совпадают с видимыми русскими подписями; ошибки, загрузка и ограничение
попыток объявляются; фокус переходит к началу нового шага; состояния
различимы без опоры только на цвет; формы работоспособны при увеличенном
тексте и в светлом, тёмном и повышенно-контрастном оформлении.
Covers: IOS-REQ-2

### IOS-AC-3 — Template and scope isolation

Сфокусированная проверка source и runtime подтверждает отсутствие прежнего
Core Data шаблона и исключённых границ (выход, сброс пароля, аналитика,
внешние провайдеры) в корневой поверхности и новом флоу.
Covers: IOS-REQ-3

### IOS-AC-4 — Gated launch without session

Запуск без валидной сессии показывает спокойное состояние загрузки, затем шаг
почты; навигация оболочки не отображается и недостижима до входа.
Covers: IOS-REQ-4

### IOS-AC-5 — Session persistence across launches

Перезапуск приложения в пределах жизни сессии открывает оболочку с активным
разделом «Кейсы» сразу, без экрана входа и без мигания формой.
Covers: IOS-REQ-4

### IOS-AC-6 — Email branching and return

Почта существующего аккаунта ведёт к шагу пароля с заголовком «Вход»; новая
почта — к шагу с заголовком «Регистрация»; возврат на шаг почты сохраняет
введённое значение доступным для правки, почта остаётся видимой на шаге
пароля.
Covers: IOS-REQ-5

### IOS-AC-7 — Success opens shell

Верные учётные данные при входе и валидная новая пара при регистрации
немедленно открывают оболочку с активным разделом «Кейсы»; возврат в флоу из
оболочки навигационно недоступен.
Covers: IOS-REQ-6

### IOS-AC-8 — Local validation before network

Пустое обязательное поле, синтаксически невалидная почта и пароль короче 6
символов при регистрации отклоняются с человекочитаемым сообщением рядом с
полем, и сетевой запрос при этом не отправляется.
Covers: IOS-REQ-7

### IOS-AC-9 — Server error states

Ответ `401` на шаге входа показывает восстановимую ошибку без раскрытия
существования аккаунта с очищенным полем пароля и доступным повтором; ответ
`409` при регистрации объясняет занятую почту и предлагает вход; ответ `422`
показывает серверную валидацию рядом с формой.
Covers: IOS-REQ-7

### IOS-AC-10 — Rate limit state

Ответ `429` с `Retry-After` переводит флоу в отдельное состояние с указанием
времени повтора; до истечения срока активация основного действия не
отправляет запрос; появление состояния объявляется ассистивно.
Covers: IOS-REQ-7

### IOS-AC-11 — Offline recovery

При отсутствии сети флоу показывает отдельное восстановимое состояние с
действием повтора; после восстановления связи повтор успешен; введённая почта
сохранена, пароль отсутствует.
Covers: IOS-REQ-7

### IOS-AC-12 — Session expiry return and cleanup

Ответ `401` на персональный запрос возвращает на шаг почты с коротким
объяснением; персональные данные прошлой сессии не отображаются, а проверка
хранилища подтверждает отсутствие недействительного сессионного секрета.
Covers: IOS-REQ-8

### IOS-AC-13 — Storage inspection

Проверка хранилища устройства после входа показывает: пароль не сохранён ни в
защищённом хранилище, ни в файлах или настройках приложения; значение
сессионного cookie находится только в защищённом системном хранилище класса
`Keychain`, недоступном другим приложениям, и не встречается в логах.
Covers: IOS-REQ-9

### IOS-AC-14 — API client observable contract

Модульные проверки клиента подтверждают: четыре операции (`email-check`,
`login`, `register`, `profile`) уходят только по `https`; единый формат
ошибок бэкенда детерминированно отображается в состояния флоу; cookie-сессия
предъявляется в персональных запросах; `401` персонального запроса
маршрутизируется в возврат на вход.
Covers: IOS-REQ-10

### IOS-AC-15 — Concurrency discipline

Проверки конкурентности подтверждают: во время активного запроса повторная
активация основного действия не порождает второй сетевой вызов; уход с шага
отменяет незавершённый запрос; отменённый запрос не меняет видимое состояние;
интерфейс остаётся отзывчивым во время загрузки.
Covers: IOS-REQ-11

### IOS-AC-16 — Physical unit boundary

Boundary-проверка подтверждает: auth-возможность собрана как отдельный
`Swift package` с собственными тестами; граф зависимостей направлен от
цели приложения к единице; публичный контракт единицы минимален; вердикт
проверки границы пакета — `PASS`.
Covers: IOS-REQ-12

### IOS-AC-17 — Parity walkthrough

Сквозной проход «почта → пароль → оболочка» на simulator демонстрирует шаги,
тексты и правила гейтинга, совпадающие с общим контрактом; отклонения
ограничены явно разрешённой нативной вариативностью.
Covers: IOS-REQ-13

## Constraints

`SysDevScenApp.swift`, `ContentView.swift` и проект `Xcode` — пути evidence, а
не разрешение предполагать конкретный SDK API, тип `SwiftUI`-контейнера или
сетевой стек: эти решения принимает `architecture-designer` в `design.md`.
Запрещены глобальный изменяемый singleton, ослабление default `ATS`,
логирование пароля или сессионного секрета, вымышленный API дизайн-системы и
расхождение с общим поведением. Нативные визуальные решения (Liquid Glass,
appearance-варианты, fallback для старых OS/SDK) задаёт `platform-ux.md`.
Контракт бэкенда фиксирован (email-first шаги, cookie-сессия 14 дней, минимум
6 символов пароля, ограничение 5 попыток на 15 минут, единый формат ошибок);
любое ограничение iOS, конфликтующее с общим поведением, возвращает работу в
product elaboration.

## Integration points

Существующий application target `SysDevScen` остаётся границей входа и
композиции; гейт запуска встраивается в его корневую маршрутизацию.
`SysDevScenTests` и `SysDevScenUITests` — цели для сфокусированных модульных и UI-проверок;
новая auth-единица получает собственные модульные тесты. Проект `Xcode`
остаётся источником истины для membership, scheme и runtime discovery
(deployment target `iOS 26.5`, симуляторы `iOS 26.5` и `iOS 18.6`). Бэкенд
доступен по `https://89.125.1.21.nip.io`; адрес — внешний параметр окружения.
Никакой source path, кроме документированного текущего корня, не объявляется
существующим заранее.
