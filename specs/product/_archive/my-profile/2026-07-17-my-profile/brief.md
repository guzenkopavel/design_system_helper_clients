# my-profile — product brief

## Problem and Users

Авторизованный пользователь уже попадает в оболочку приложения, но вкладка
профиля остаётся без полезного поведения: нельзя увидеть текущий аккаунт,
проверить наличие собственных интервью и явно выйти из сессии. Это мешает
контролировать персональный контекст на устройстве и закрывает простой путь к
смене аккаунта.

Целевые пользователи — авторизованные пользователи iOS и Android клиентов,
которые открывают одну из трёх вкладок приложения — профиль — и ожидают увидеть
свой аккаунт, связанную историю интервью и действие выхода.

## Current Baseline

У `my-profile` пока нет собственного доставленного
`specs/product/my-profile/SPECIFICATION.md`.

Зависимый baseline из `specs/product/user-profile-auth/SPECIFICATION.md`
сохраняется как входной продуктовый контракт авторизованного состояния:

- без валидной сессии приложение показывает только флоу авторизации;
- успешные вход и регистрация открывают оболочку;
- валидная серверная сессия сохраняет вход между запусками;
- истёкшая или отозванная сессия возвращает пользователя на экран входа и
  очищает недействительный сессионный секрет;
- тексты auth-флоу фиксированы на русском, поведение одинаково для iOS и
  Android.

Новое поведение `my-profile`:

- первая профильная вкладка получает видимый контент текущего аккаунта,
  действие «Мои интервью» и явный выход;
- после выхода пользователь возвращается на экран ввода почты с пустым полем
  почты.

## Outcome and Success Signals

- Пользователь понимает, под какой почтой он вошёл, без перехода в другие
  разделы.
- Пользователь может проверить количество своих интервью из профиля одним
  действием.
- При отсутствии интервью действие «Мои интервью» визуально и семантически
  недоступно.
- Выход завершает текущую сессию и возвращает пользователя к чистому экрану
  ввода почты.

## Evidence and Assumptions

| Утверждение | Доказательство или проверка | Уверенность |
|---|---|---|
| Текущий профиль содержит `email`, который можно показывать пользователю. | Документация OpenAPI для `GET /api/profile`: `ProfileResponse` содержит обязательные `id` и `email`. | Высокая |
| История интервью принадлежит текущему профилю и доступна по серверной сессии. | Документация OpenAPI для `GET /api/interviews/history`: описание фиксирует историю текущего профиля, защиту `SessionCookie` и ответ `InterviewHistoryListResponse.data`. | Высокая |
| Количество интервью для кнопки и сообщения можно определить после загрузки всех страниц истории. | `InterviewHistoryListResponse.data` — массив `InterviewHistoryItemResponse`; `page.hasMore` и `nextCursor` позволяют продолжать чтение страниц, прямого поля общего количества в схеме нет. | Высокая |
| История может быть постраничной, поэтому первый ответ не доказывает полный счётчик при `page.hasMore = true`. | `InterviewHistoryListResponse.page` содержит `hasMore` и `nextCursor`; параметр `limit` по умолчанию 20. | Высокая |
| Выход серверно очищает сессию и пригоден для явного завершения входа. | Документация OpenAPI для `POST /api/auth/logout`: `204`, очищает `SessionCookie`, возвращает успех для активной, отсутствующей или недействительной сессии. | Высокая |
| Экран профиля находится в одной из трёх вкладок оболочки. | Ввод пользователя: “У нас есть 3 вкладки. И вот мы сейчас про 1 из них.” | Высокая |

## Chosen Direction

### Product decisions

- Вкладка профиля показывает крупную стандартную иконку профиля по центру с
  небольшим верхним отступом, затем email текущего пользователя.
- Ниже расположен единый блок с двумя действиями: «Мои интервью» и «Выход».
- «Мои интервью» кликабельна только когда у пользователя есть минимум одно
  интервью; при нажатии показывается toast с количеством интервью.
- Если интервью нет, «Мои интервью» остаётся видимой, но недоступной для
  активации.
- Количество интервью для toast считается по всей загруженной истории текущего
  профиля; если история постраничная, клиент догружает страницы до завершения
  пагинации перед включением действия.
- «Выход» всегда доступен авторизованному пользователю; успешный выход
  завершает сессию и возвращает на первый шаг auth-флоу с пустой почтой.
- Если при запросе профиля или истории сервер отвечает, что сессия
  недействительна, применяется базовое восстановление авторизации: возврат к auth-флоу без
  персональных данных.

### Rejected alternatives

- Скрывать «Мои интервью» при нулевом количестве: отклонено, потому что
  постоянная видимость лучше объясняет наличие будущей персональной истории.
- Открывать отдельный экран списка интервью: отклонено для этого scope;
  пользователь запросил toast с количеством, а не навигацию в историю.
- Показывать полный профиль с именем, настройками или аватаром пользователя:
  отклонено, потому что доступный профильный контракт содержит только email и
  технический `id`.

## Scope

- Наполнение вкладки профиля для авторизованного состояния.
- Отображение стандартной крупной иконки профиля, email текущего пользователя
  и блока действий.
- Получение истории интервью текущего профиля для определения доступности
  «Мои интервью» и количества для toast.
- Явный выход из сессии через профиль и возврат на экран ввода почты с пустым
  значением.
- Состояния загрузки, пустой истории, ошибки истории, недействительной сессии,
  доступности, локализации, приватности и паритета для iOS/Android на уровне
  наблюдаемого поведения.

## Non-Goals

- Экран списка интервью, детали интервью, продолжение интервью или фильтры
  истории.
- Редактирование профиля, загрузка пользовательского аватара, отображение имени
  или иных персональных полей сверх email.
- Изменение auth API, backend API, веб-клиента или схемы истории интервью.
- Изменение двух остальных вкладок оболочки.
- Аналитика событий профиля.
- Платформенная архитектура, SDK-компоненты, конкретные иконки библиотек или
  способ хранения сессии.

## Shared Observable Behavior

- Авторизованный пользователь открывает вкладку профиля и видит крупную
  стандартную иконку профиля, свою почту и блок с действиями «Мои интервью» и
  «Выход».
- До завершения загрузки данных профиля/истории персональный контент не должен
  мигать неконсистентными значениями; действия, зависящие от данных истории,
  не допускают повторных или преждевременных активаций.
- Если история содержит минимум один элемент, «Мои интервью» доступна; нажатие
  показывает toast: «Интервью: N», где `N` — известное приложению количество
  интервью.
- Если история пуста, «Мои интервью» недоступна для нажатия и имеет
  различимое disabled-состояние.
- Если история постраничная, приложение догружает страницы до завершения
  пагинации и считает все полученные элементы; до завершения этой загрузки
  «Мои интервью» не активируется.
- «Выход» вызывает завершение сессии; после успешного выхода пользователь
  попадает на экран ввода почты, поле почты пустое, персональные данные профиля
  и истории не отображаются.
- Ошибка загрузки истории не должна блокировать выход. Пока история недоступна,
  «Мои интервью» не активируется, а пользователю показывается восстановимое
  состояние или возможность повторить загрузку в рамках профиля.
- Недействительная сессия при запросах профиля/истории/выхода возвращает к
  базовому восстановлению auth-флоу без отображения персональных данных прошлой сессии.

## Draft Journey and Flow Impact

### Точки входа и выхода

- Вход: авторизованный пользователь выбирает вкладку профиля в существующей
  оболочке из трёх вкладок.
- Выход: пользователь остаётся в профиле после просмотра сообщения или уходит в
  auth-флоу после «Выход».

### Affected or new surfaces

- Вкладка профиля в авторизованной оболочке.
- Краткое сообщение для действия «Мои интервью».
- Первый экран auth-флоу после выхода, где поле почты пустое.

### Draft screen/flow map

1. Профиль открывается в авторизованном состоянии.
2. Экран показывает верхний центрированный профильный символ.
3. Под символом отображается email текущего профиля.
4. Ниже расположен блок действий:
   - «Мои интервью»;
   - действие «Выход».
5. «Мои интервью»:
   - `count > 0` → действие доступно → нажатие → сообщение с количеством;
   - `count = 0` → действие недоступно.
6. «Выход» → завершение сессии → auth-флоу → экран ввода почты с пустым полем.

### States, navigation and content semantics

- **Загрузка:** профиль и история загружаются спокойно; действие «Мои интервью»
  не активируется до известного состояния истории.
- **Пустая история:** история интервью пуста; «Мои интервью» видима, но недоступна.
- **История доступна:** история содержит интервью; «Мои интервью» доступна и сообщает
  количество через краткое сообщение.
- **Ошибка:** ошибка истории объясняется восстановимо; выход остаётся доступным.
- **Нет сети:** если историю нельзя загрузить из-за сети, профиль показывает
  восстановимое состояние; «Мои интервью» недоступна до успешной загрузки.
- **Нет авторизации:** недействительная сессия возвращает на auth-флоу по базовому контракту.
- **Загрузка выхода:** во время выхода повторная активация «Выход» невозможна,
  состояние занятости видимо.
- **Успешный выход:** экран ввода почты открыт, поле почты пустое.
- **Ошибка выхода:** если выход не завершён из-за сетевой ошибки, пользователь
  остаётся в профиле с восстановимым сообщением; персональные данные не должны
  очищаться локально так, чтобы создать ложное состояние выхода.

### Accessibility and localization

- Все видимые тексты на русском: «Мои интервью», «Выход», сообщение с количеством.
- Email доступен ассистивно как email текущего профиля.
- Иконка профиля имеет семантику декоративного/профильного символа без
  дублирования email как имени кнопки.
- Недоступное состояние «Мои интервью» различимо не только цветом и объявляется
  ассистивно как недоступное.
- Краткое сообщение объявляет количество интервью
  ассистивно.
- Экран поддерживает увеличенный текст, светлое, тёмное и
  повышенно-контрастное оформление без перекрытий.

### Analytics and privacy

- Новая аналитика не добавляется.
- Email отображается только в авторизованном профиле и очищается с экрана после
  выхода или недействительной сессии.
- История интервью используется только для доступности действия и количества в
  кратком сообщении; содержимое интервью на этом экране не раскрывается.
- Сессионные данные и запросы продолжают следовать базовому privacy/security
  контракту.

### Shared design-system intent

- Визуальное направление остаётся спокойным soft-blue через смысловые роли:
  основной интерактивный акцент для доступного действия, критическая роль
  для «Выход», приглушённая роль для недоступных «Моих интервью»,
  тихая нейтральная поверхность для блока действий.
- Композиция простая и центрированная в верхней части профиля: крупный
  профильный символ, email как идентификатор аккаунта, затем компактный блок
  действий.
- Состояния различимы в светлом, тёмном и повышенно-контрастном оформлении и
  используют признаки не только цветом.
- Платформы могут адаптировать стандартный профильный символ и краткое
  сообщение нативно, сохраняя общий наблюдаемый intent.

## Draft Requirements

- `REQ-1` — Авторизованная вкладка профиля отображает текущий аккаунт через
  крупную стандартную профильную иконку, email пользователя и блок действий
  «Мои интервью» / «Выход».
- `REQ-2` — Действие «Мои интервью» доступно только при наличии минимум одного
  интервью у текущего профиля.
- `REQ-3` — Нажатие доступного действия «Мои интервью» показывает краткое
  сообщение с количеством интервью текущего профиля.
- `REQ-4` — При отсутствии интервью действие «Мои интервью» остаётся видимым,
  но недоступным для активации.
- `REQ-5` — Действие «Выход» завершает текущую сессию и возвращает пользователя
  на экран ввода почты с пустым полем почты.
- `REQ-6` — Ошибки загрузки истории, отсутствие сети и недействительная сессия
  отображаются как восстановимые или базовые recovery-состояния без
  раскрытия персональных данных после потери авторизации.
- `REQ-7` — Поведение профиля, доступности действий, toast-сообщения и выхода
  одинаково для iOS и Android на уровне наблюдаемого продукта.
- `REQ-8` — Экран профиля доступен ассистивно, поддерживает увеличенный текст,
  светлое, тёмное и повышенно-контрастное оформление, а недоступные состояния и feedback
  состояния различимы не только цветом.

## Draft Acceptance Criteria

- `AC-1` — При открытии профиля авторизованный пользователь видит крупную
  профильную иконку, свою почту и блок с действиями «Мои интервью» и «Выход».
  `Covers: REQ-1`
- `AC-2` — Если история текущего профиля содержит хотя бы одно интервью,
  «Мои интервью» доступна для активации. `Covers: REQ-2`
- `AC-3` — Нажатие доступной «Мои интервью» показывает toast или эквивалентное
  краткое сообщение «Интервью: N» с количеством интервью. `Covers: REQ-3`
- `AC-4` — Если история текущего профиля пуста, «Мои интервью» видима, но
  недоступна для нажатия. `Covers: REQ-4`
- `AC-5` — Нажатие «Выход» при успешном logout возвращает на экран ввода почты,
  и поле почты пустое. `Covers: REQ-5`
- `AC-6` — После успешного выхода email и данные истории прошлого профиля не
  отображаются в оболочке. `Covers: REQ-5`
- `AC-7` — При ошибке загрузки истории или офлайн-состоянии «Выход» остаётся
  доступным, а «Мои интервью» не активируется до успешного определения истории.
  `Covers: REQ-6`
- `AC-8` — Ответ о недействительной сессии при загрузке профиля или истории
  возвращает пользователя в auth-флоу по базовому контракту без отображения персональных
  данных прошлой сессии. `Covers: REQ-6`
- `AC-9` — Один и тот же набор состояний профиля, правил доступности,
  текста сообщения и результата выхода наблюдается на iOS и Android.
  `Covers: REQ-7`
- `AC-10` — Ассистивная проверка объявляет email, доступность или
  недоступность «Моих интервью», сообщение с количеством и занятость выхода.
  `Covers: REQ-8`
- `AC-11` — При увеличенном тексте, светлом, тёмном и повышенно-контрастном
  оформлении контент профиля остаётся читаемым, не перекрывается, а недоступное
  состояние различимо без опоры только на цвет. `Covers: REQ-8`

## Product Constraints

- API и backend-контракт не меняются этим изменением.
- История интервью берётся из `GET /api/interviews/history` для текущей
  серверной сессии; endpoint защищён `SessionCookie`.
- Выход использует `POST /api/auth/logout`, который очищает `SessionCookie` и
  возвращает `204`.
- Прямого поля общего количества интервью в обнаруженной схеме нет; полный
  счётчик определяется догрузкой страниц истории до `page.hasMore = false`.
- Адрес API остаётся внешним параметром окружения, не продуктовым решением.
- Новые платформенные implementation specs не создаются на Discovery.

## Risks

- Догрузка всех страниц истории может быть медленнее на аккаунтах с большой
  историей; UX должен сохранять спокойное loading-состояние и доступный выход.
- Сетевой сбой logout может создать риск ложного ощущения выхода; требуется
  явное восстановимое состояние.
- Disabled-кнопка «Мои интервью» без пояснения может быть непонятной
  пользователю с нулевой историей; UX/elaborate должен решить, нужно ли
  вторичное пояснение без расширения scope.
- Email в профиле является персональными данными; после выхода или invalid
  session он не должен оставаться на экране.
