## Problem and Why

Оба мобильных клиента открываются в доставленной оболочке `app-shell` без
идентификации пользователя, поэтому персональные возможности бэкенда Design
System Helper (интервью, история, профиль) им недоступны. Бэкенд уже
предоставляет email-first авторизацию с серверной сессией; веб-клиент по ней
работает. Мобильным клиентам нужен тот же вход, чтобы будущие фичи получили
аутентифицированный контекст, а пользователь не вводил учётные данные при
каждом запуске.

## Outcomes and Success Signals

- Новый пользователь проходит путь «почта → пароль → оболочка» без внешних
  инструкций.
- Повторный запуск в пределах жизни серверной сессии открывает оболочку
  сразу, без экрана входа.
- Ошибки флоу (валидация, неверные данные, ограничение попыток, офлайн)
  объяснены и восстановимы; пользователь не застревает.

## Scope

- Общий email-first флоу входа и регистрации для iOS и Android поверх
  существующего API (`email-check`, `login`, `register`, `profile`).
- Гейт всего приложения: до входа — только флоу авторизации; после входа —
  доставленная оболочка `app-shell`.
- Сохранение аутентифицированного состояния между запусками на срок жизни
  серверной сессии; определение состояния при запуске.
- Наблюдаемые состояния флоу: загрузка, локальная валидация, неверные учётные
  данные, занятая почта, ограничение попыток с временем повтора, отсутствие
  сети, истёкшая или отозванная сессия.

## Non-Goals

- Пользовательский выход («Выйти») и любое наполнение раздела «Профиль» —
  отдельная будущая фича; в этой фиче сессию завершает только её истечение
  или отзыв на сервере.
- Сброс пароля, подтверждение почты, внешние провайдеры (отсутствуют на
  бэкенде сознательно).
- Продление сессии, биометрический вход, PIN-код.
- Офлайн-кэш контента и любые персональные данные в оболочке.
- Реальное содержимое разделов «Кейсы», «Знания», «Профиль».
- Изменение поведения веба и бэкенда.
- Предписание платформенных компонентов, SDK, API хранения или архитектуры.

## Product Decisions

- Email-first вход по контракту бэкенда: почта → проверка → явный шаг «Вход»
  либо «Регистрация» → оболочка. Ручной выбор ветки отклонён: сервер уже
  решает ветку, ручной выбор добавляет ошибки и расходится с вебом.
- Всё приложение за авторизацией. Паритет с вебом (анонимные «Знания») и
  вход по требованию отклонены решением владельца продукта.
- Сессия сохраняется между запусками на срок её жизни (до 14 дней по
  контракту бэкенда); при валидной сессии вход не запрашивается.
- Все три поверхности оболочки остаются baseline-заглушками; выход и профиль
  вынесены в будущую фичу. Принятое следствие: сменить аккаунт на устройстве
  нельзя до истечения или отзыва сессии.
- Пустое обязательное поле не отправляется: попытка продолжить показывает
  человекочитаемое объяснение рядом с полем; основное действие остаётся
  видимым и доступным для активации (единое поведение обоих клиентов).
- Недействительный (истёкший или отозванный) сессионный секрет очищается из
  хранилища устройства при возврате на экран входа.
- Поведение системного возврата на первом шаге — стандартное для платформы;
  флоу не перехватывает его нестандартно.
- Системное автозаполнение учётных данных допускается как нативная
  возможность, но не является обязательством этой фичи.

## Shared Observable Behavior

### Happy path

1. Запуск: приложение определяет состояние сессии, показывая спокойное
   состояние загрузки без мигания формой входа.
2. Валидная сессия → оболочка с активным разделом «Кейсы». Иначе → шаг почты.
3. Пользователь вводит почту и продолжает; по ответу проверки открывается шаг
   пароля с заголовком «Вход» (аккаунт существует) либо «Регистрация» (почта
   новая); почта остаётся видимой, возврат к её правке доступен.
4. Верный пароль (вход) или валидная новая пара (регистрация) немедленно
   открывает оболочку с активным разделом «Кейсы».
5. Повторные запуски в пределах жизни сессии открывают оболочку сразу.

### Secondary states

- **Loading:** во время любого запроса основное действие шага заблокировано с
  видимым индикатором; повторная отправка невозможна.
- **Empty:** пустая почта или пустой пароль не отправляются; попытка
  продолжить показывает человекочитаемое объяснение рядом с полем.
- **Валидация:** синтаксически невалидная почта и пароль короче 6 символов
  при регистрации отклоняются до сетевого запроса с человекочитаемым
  сообщением.
- **Error:** неверные учётные данные сообщаются без раскрытия существования
  аккаунта; поле пароля очищается, повтор доступен сразу. Занятая почта при
  регистрации объясняет, что аккаунт уже есть, и предлагает вход.
- **Rate limit:** ограничение попыток — отдельное состояние с указанием
  времени повтора; до истечения срока запрос не отправляется; появление
  состояния объявляется ассистивно.
- **Offline:** отсутствие сети — отдельное восстановимое состояние с
  возможностью повторить; введённая почта сохраняется, пароль — нет.
- **Recovery:** истёкшая или отозванная сессия возвращает на шаг почты с
  коротким объяснением; персональные данные прошлой сессии не отображаются,
  недействительный сессионный секрет очищается из хранилища устройства.
- **Accessibility:** имена элементов совпадают с видимыми русскими подписями;
  ошибки, загрузка и ограничение попыток объявляются ассистивно; смена шага
  переводит фокус к логическому началу нового шага; состояния различимы не
  только цветом; поддержаны увеличенный текст, светлое, тёмное и
  повышенно-контрастное оформление.
- **Localization:** тексты фиксированы на русском текущим контрактом.
- **Analytics/privacy:** аналитика не добавляется; пароль не сохраняется и не
  логируется; сессионные данные хранятся недоступно другим приложениям;
  учётные и сессионные данные передаются только по защищённому каналу.

## Requirements

- `REQ-1` — Без валидной сессии приложение показывает только флоу
  авторизации; разделы оболочки недоступны до входа.
- `REQ-2` — Флоу авторизации начинается с ввода почты; по результату проверки
  пользователь явно видит, выполняет он вход или регистрацию для этой почты,
  и может вернуться к изменению почты без потери введённого значения.
- `REQ-3` — Успешные вход и регистрация немедленно открывают оболочку со
  стартовым разделом «Кейсы».
- `REQ-4` — Аутентифицированное состояние сохраняется между запусками
  приложения на срок жизни серверной сессии; при валидной сессии вход не
  запрашивается повторно.
- `REQ-5` — Ошибки флоу (пустые поля, невалидная почта, короткий пароль при
  регистрации, неверные учётные данные, занятая почта, ограничение попыток,
  отсутствие сети) отображаются как различимые понятные восстановимые
  состояния; ограничение попыток сообщает время повтора.
- `REQ-6` — Истечение или отзыв сессии возвращает пользователя на экран входа
  с объяснением; персональные данные прошлой сессии перестают отображаться, а
  недействительный сессионный секрет очищается из хранилища устройства.
- `REQ-7` — Пароль не сохраняется на устройстве; сессионные данные хранятся
  недоступно другим приложениям.
- `REQ-8` — Флоу авторизации доступен ассистивно и в согласованных вариантах
  оформления: имена совпадают с видимыми подписями, ошибки, загрузка и
  ограничение попыток объявляются, фокус управляется при смене шага,
  состояния различимы не только цветом, поддержаны увеличенный текст,
  светлое, тёмное и повышенно-контрастное оформление.
- `REQ-9` — Наблюдаемое поведение флоу, тексты шагов и правила гейтинга
  одинаковы на iOS и Android.

## Acceptance Criteria

- `AC-1` — Запуск без валидной сессии открывает экран входа с полем почты, и навигация оболочки не отображается. `Covers: REQ-1` `Verification dimension: launch-gating`
- `AC-2` — Ввод почты существующего аккаунта ведёт к шагу пароля с заголовком «Вход». `Covers: REQ-2` `Verification dimension: email-branching-login`
- `AC-3` — Ввод почты без аккаунта ведёт к шагу пароля с заголовком «Регистрация». `Covers: REQ-2` `Verification dimension: email-branching-register`
- `AC-4` — Возврат с шага пароля к шагу почты сохраняет введённую почту доступной для правки. `Covers: REQ-2` `Verification dimension: email-step-return`
- `AC-5` — Верные учётные данные существующего аккаунта открывают оболочку с активным разделом «Кейсы». `Covers: REQ-3` `Verification dimension: login-success-entry`
- `AC-6` — Валидная новая пара почта/пароль на шаге регистрации создаёт аккаунт и открывает оболочку с активным разделом «Кейсы». `Covers: REQ-3` `Verification dimension: register-success-entry`
- `AC-7` — Перезапуск приложения в пределах жизни сессии открывает оболочку без экрана входа. `Covers: REQ-4` `Verification dimension: session-persistence`
- `AC-8` — Неверный пароль показывает восстановимую ошибку без раскрытия существования аккаунта, поле пароля очищено и повтор доступен. `Covers: REQ-5` `Verification dimension: invalid-credentials`
- `AC-9` — Попытка продолжить с пустым обязательным полем не отправляет запрос и показывает человекочитаемое объяснение рядом с полем. `Covers: REQ-5` `Verification dimension: empty-field-validation`
- `AC-10` — Синтаксически невалидная почта отклоняется до сетевого запроса с понятным сообщением. `Covers: REQ-5` `Verification dimension: local-email-validation`
- `AC-11` — Пароль короче 6 символов при регистрации отклоняется до сетевого запроса с понятным сообщением. `Covers: REQ-5` `Verification dimension: local-password-validation`
- `AC-12` — Ответ об ограничении попыток переводит флоу в состояние с указанием времени повтора, и до истечения срока основное действие не отправляет запрос. `Covers: REQ-5` `Verification dimension: rate-limit-state`
- `AC-13` — Попытка регистрации на занятую почту показывает объяснение, что аккаунт уже существует, с предложением выполнить вход. `Covers: REQ-5` `Verification dimension: registered-email-conflict`
- `AC-14` — При отсутствии сети флоу показывает отдельное восстановимое состояние с возможностью повторить после восстановления связи. `Covers: REQ-5` `Verification dimension: offline-recovery`
- `AC-15` — Истечение или отзыв сессии при персональном запросе возвращает на экран входа с объяснением, и данные прошлой сессии не отображаются. `Covers: REQ-6` `Verification dimension: session-expiry-return`
- `AC-16` — После возврата на экран входа по истечению или отзыву сессии недействительный сессионный секрет отсутствует в хранилище устройства. `Covers: REQ-6` `Verification dimension: stale-session-cleanup`
- `AC-17` — Проверка хранилища устройства после входа показывает отсутствие сохранённого пароля. `Covers: REQ-7` `Verification dimension: password-not-stored`
- `AC-18` — Сессионные данные размещены в хранилище, недоступном другим приложениям. `Covers: REQ-7` `Verification dimension: session-storage-isolation`
- `AC-19` — Ассистивная проверка флоу объявляет имена полей и действий, совпадающие с видимыми подписями, а также появление ошибок, состояний загрузки и ограничения попыток. `Covers: REQ-8` `Verification dimension: assistive-semantics`
- `AC-20` — При смене шага ассистивный фокус переходит к логическому началу нового шага. `Covers: REQ-8` `Verification dimension: focus-management`
- `AC-21` — Все состояния флоу (ошибка, загрузка, заблокированность, выбранный шаг) различимы без опоры только на цвет. `Covers: REQ-8` `Verification dimension: non-color-cues`
- `AC-22` — При увеличенном тексте формы флоу остаются читаемыми и работоспособными. `Covers: REQ-8` `Verification dimension: text-scaling`
- `AC-23` — В светлом и тёмном оформлении подписи, сообщения и состояния полей сохраняют читаемость. `Covers: REQ-8` `Verification dimension: light-dark-appearance`
- `AC-24` — В повышенно-контрастном оформлении подписи, сообщения и состояния полей сохраняют читаемость и различимость. `Covers: REQ-8` `Verification dimension: increased-contrast`
- `AC-25` — Сквозной проход флоу (почта → пароль → оболочка) на iOS и Android демонстрирует одинаковые шаги, тексты и правила гейтинга. `Covers: REQ-9` `Verification dimension: cross-client-parity`

## Common Product Constraints

- Контракт бэкенда фиксирован и не меняется этой фичей: email-first шаги,
  серверная cookie-сессия сроком 14 дней без продления, минимальная длина
  пароля 6 символов, ограничение 5 попыток на 15 минут с указанием времени
  повтора, единый формат ошибок с кодом, сообщением и идентификатором
  трассировки.
- Учётные данные и сессионный секрет передаются только по защищённому
  каналу; передача по незащищённому соединению недопустима.
- Проверка почты раскрывает существование аккаунта по контракту бэкенда;
  клиенты не усиливают это раскрытие.
- Baseline `app-shell` остаётся источником поведения оболочки; эта фича
  меняет только гейтинг запуска.
- Адрес API — внешний параметр окружения, не продуктовое решение.
