## Goal

Отрисовать экраны авторизации с материальными компонентами, русскими подписями из
ресурсов, состояниями загрузки и ошибок, ветвлением входа или регистрации,
доступной семантикой и запасным вариантом мягко-синего цвета из документа
пользовательского опыта.

## Inline contract context

`AND-REQ-4` требует процесс с явным заголовком входа или регистрации.
`AND-REQ-5` требует переход в оболочку при успехе. `AND-REQ-7` требует
различимые состояния для каждого наблюдаемого сценария ошибки. `AND-REQ-10`
требует материальные компоненты, русские строковые ресурсы, имена ассистивных
технологий, управление фокусом, нецветовые индикаторы, масштабируемый текст и
поддержку светлого, тёмного и повышенно-контрастного оформления. `AND-AC-5`
требует экран входа без сессии и оболочку с сессией. `AND-AC-6` требует заголовок
входа для существующей почты. `AND-AC-7` требует заголовок регистрации для новой
почты. `AND-AC-8` требует сохранение введённой почты при возврате.

## Implementation deliverables

- Появится компонуемый экран авторизации с двухшаговым процессом: шаг ввода
  почты с оформленным полем и заливочной кнопкой, шаг ввода пароля с явным
  заголовком, индикаторами загрузки, сообщениями ошибок и кнопками возврата;
  модель представления определяет текущий шаг и загруженность, а состояние
  передаётся однонаправленно через компонуемый фреймворк.
- Появятся русские строковые ресурсы, покрывающие все подписи полей, кнопки,
  ошибки, состояния загрузки и ограничения попыток, совпадающие с видимыми и
  доступными именами ассистивных технологий; семантические роли материального
  интерфейса обеспечивают корректные контрасты в светлом и тёмном оформлении
  с мягко-синим запасным вариантом.

## Steps

До реализации написать тесты интерфейса для шага почты с валидацией, шага пароля
с заголовком входа, шага пароля с заголовком регистрации, индикатора загрузки в
кнопке, сообщения ошибки рядом с полем, сохранения почты при возврате, состояния
ограничения попыток с заблокированной кнопкой и имён ассистивных технологий.
Реализовать компонуемый элемент с вынесенным состоянием на материальном интерфейсе,
сохраняя состояние однонаправленным и стабильным. Подписи хранить в ресурсах,
а запасной вариант локализации проверять с учётом длинного текста. Выполнить
проверки дизайн-системы через семантические роли цветов, типографику, формы и
нецветовые признаки выбора. Следовать документу пользовательского опыта:
material interface выбран, light и dark оформление должно раскрывать доступные
цветовые роли, динамические цвета остаются выключенными, мягко-синий запасной
вариант авторитетен; выразительные расширения исключены. Проверка эмулятора
и доступности добавляется при обнаружении runtime.

## Verification

Запустить фокусированные сценарии компонуемого фреймворка. Если эмулятор или
устройство недоступен, только runtime доказательства записываются как неизвестные.
Проверка на эмуляторе — маркер emulator зафиксирован. Проверка доступности — маркер
accessibility подтверждён. Проверка дизайн-системы — маркер design-system выполнен.
Документ platform-ux.md является источником проверок. Доказательства подтверждают
module boundary и module build. Публичный контракт виден — это public contract.
Потребитель интегрирован — маркер consumer integration. Зависимости верны — маркер
dependency graph. Подключение оболочки завершено — маркер app-shell wiring.
Интерфейс следует Material 3. Оформление поддерживает light/dark режимы. Цветовые
роли доступны — это accessible on-colors. Проверка dynamic color подтверждена.
Запасной вариант мягко-синего — маркер soft-blue fallback.

## Expected result

Модуль авторизации содержит компонуемый экран с материальными компонентами,
русскими подписями, доступной семантикой, корректными переходами между шагами,
обработкой всех состояний ошибок и согласованной поддержкой светлого, тёмного
оформления с мягко-синим запасным вариантом. Результат сохраняет понятный
интерфейс, русские подписи, базовую платформу визуализации и доступность
без неподтверждённых расширений.

## Out of scope

Настройка приложения, маршрутизация шлюза, изменения главного файла, тесты
хранилища в runtime, включение динамических цветов, внедрение выразительного
оформления, реализация сетевого транспорта и контент продукта в эту задачу не
входят.
