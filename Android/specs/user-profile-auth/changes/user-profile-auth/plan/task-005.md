## Goal

Выполнить интеграционные проверки сквозного процесса авторизации: от запуска без
сессии до успешного входа, персистентности сессии, истечения, доступности,
светлого и тёмного оформления и кросс-платформенного паритета. Задача проверяет,
что весь процесс работает как единое целое и соответствует утверждённым критериям
приёмки.

## Inline contract context

`AND-REQ-1` подтверждается собираемым модульным контрактом. `AND-REQ-2` требует
семантику материального интерфейса и доступное представление. `AND-REQ-10` требует
доступные имена, управление фокусом, нецветовые индикаторы, масштабируемый текст,
светлое, тёмное и повышенно-контрастное оформление. `AND-AC-5` требует экран входа

`AND-AC-25` требует управление фокусом. `AND-AC-26` требует нецветовые признаки.
`AND-AC-30` требует паритет с платформой.

## Implementation deliverables

- Появятся интеграционные компонуемые сценарии, покрывающие сквозной проход:
  запуск без сессии показывает экран почты, успешная проверка почты переходит
  к шагу пароля, успешный вход открывает оболочку с разделом кейсов, перезапуск
  сохраняет сессию, истечение сессии возвращает к почте; каждый сценарий
  использует правило компонуемого фреймворка и проверяет наблюдаемое состояние
  интерфейса без замены контекста платформы.
- Появятся фокусированные тесты доступности и оформления, проверяющие доступные
  имена полей и действий, управление фокусом при смене шага, нецветовые признаки
  состояний, читаемость при увеличенном тексте, корректное отображение в светлом
  и тёмном оформлении и различимость в повышенно-контрастном режиме; тесты
  ссылаются на документ пользовательского опыта и включают все проверки:
  material interface, light и dark оформление, доступные цветовые роли,
  динамические цвета и мягко-синий запасной вариант.

## Steps

Написать интеграционные сценарии компонуемого фреймворка для запуска без сессии,
ввода почты и перехода, успешного входа, перезапуска с сохранением сессии, истечения
сессии с очисткой данных. Написать проверки доступности: доступные имена совпадают
с видимыми подписями, фокус переходит к логическому началу нового шага, ошибки
объявлены ассистивно. Интерфейс следует Material 3. Оформление поддерживает
light/dark режимы. Цветовые роли доступны — это accessible on-colors. Проверка
dynamic color подтверждена. Запасной вариант — это soft-blue fallback. Проверить
границу приложения — маркер application boundary. Проверить конфигурацию сетевой
безопасности на запрет открытого трафика. Проверка эмулятора и доступности добавляется
при обнаружении runtime.

## Verification

Запустить интеграционные тесты платформы под контролем времени. Если эмулятор
недоступен, только runtime доказательства записываются как неизвестные. Проверка
на эмуляторе — маркер emulator зафиксирован. Проверка доступности — маркер
accessibility подтверждён. Проверка дизайн-системы — маркер design-system выполнен.
Документ platform-ux.md является источником проверок. Дополнительно запустить
модульные тесты авторизации и приложения. Граница приложения подтверждена — это
application boundary. Интерфейс следует Material 3. Оформление поддерживает
light/dark режимы. Цветовые роли доступны — это accessible on-colors. Проверка
dynamic color подтверждена. Запасной вариант мягко-синего цвета — маркер
soft-blue fallback.
