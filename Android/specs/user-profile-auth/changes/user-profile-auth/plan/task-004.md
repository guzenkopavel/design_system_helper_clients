## Goal

Создать корневой маршрутизатор шлюза авторизации в модуле библиотеки,
подключить главный файл к публичному контракту авторизации, настроить сетевую
безопасность приложения и сохранить модуль приложения ограниченным жизненным
циклом, корневой композицией и интеграцией потребителя.

## Inline contract context

`AND-REQ-3` требует, чтобы главный файл определял стартовое состояние — проверку
сессии и маршрутизацию между флоу авторизации и оболочкой. `AND-REQ-5` требует
немедленный переход в оболочку при валидной сессии. `AND-REQ-6` требует проверку
сохранённого секрета при запуске. `AND-REQ-9` требует защищённый канал и запрет
открытого трафика. `AND-AC-1` требует собираемую границу и одностороннюю
композицию. `AND-AC-4` требует модульную границу авторизации. `AND-AC-5` требует
экранный гейтинг без сессии и оболочку с сессией.

## Implementation deliverables

- Появится компонуемый шлюз авторизации в модуле библиотеки, выполняющий проверку
  сессионного секрета при старте: при валидной сессии немедленно показывает
  оболочку приложения, при невалидной — экран авторизации с шагом почты; при
  ошибке неавторизованного доступа возвращает на шаг почты с очисткой данных
  и сессионного секрета; компонент принимает обратный вызов при успешной
  проверке и остаётся публичной границей без раскрытия внутренней логики.
- Появится интеграция приложения с модулем авторизации: зависимость в файле
  сборки, замена безусловной оболочки в главном файле на шлюз авторизации,
  конфигурация сетевой безопасности с запретом открытого трафика, подключённая
  к манифесту; приложение сохраняет ответственность только за композицию и
  жизненный цикл, а логика авторизации остаётся в модуле-библиотеке.

## Steps

Реализовать шлюз авторизации в модуле с проверкой сессии через репозиторий: при
запуске читать сохранённый сессионный секрет, при валидной сессии компонуется
оболочка приложения, при невалидной — экран авторизации. Заменить корневой
компонент в главном файле так, чтобы он компонуется шлюз авторизации вместо
прямого вызова оболочки; сохранить тему приложения и полноэкранный режим.
Добавить зависимость модуля авторизации в файл сборки приложения. Создать
конфигурацию сетевой безопасности с запретом открытого трафика, подключить её
к манифесту. Состояние передаётся через публичный контракт, а ответственность
границы приложения ограничивается жизненным циклом и корневой композицией.
Граница модуля подтверждается: модуль авторизации не зависит от приложения и
оболочки.

## Expected result

Запуск через приложение компонуется шлюз авторизации, который маршрутизирует
между экраном авторизации и оболочкой на основе сессии; сборка потребителя
проходит; конфигурация сетевой безопасности запрещает открытый трафик; приложение
не содержит логики авторизации; фокусированные проверки покрывают маршрутизацию
и сборку. Итог задачи готовит пакет к последующей интеграционной проверке.
