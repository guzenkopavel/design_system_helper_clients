## Goal

Определить доменную модель аутентификации, реализовать сетевой транспорт,
защищённое хранилище сессии и локальную валидацию входных данных, покрыть
фокусированными модульными тестами. Задача описывает только доменную логику,
данные и хранилище авторизации без привязки к компонуемому представлению.

## Inline contract context

`AND-REQ-4` требует процесс почты и пароля с проверкой почты и ветвлением входа
или регистрации. `AND-REQ-5` требует немедленный переход в оболочку после
успешной аутентификации. `AND-REQ-6` требует сохранение и восстановление сессии.
`AND-REQ-7` требует обработку ошибок: пустые поля, невалидная почта, короткий
пароль, неверные данные, ограничение попыток, отсутствие сети. `AND-REQ-8`
требует восстановление при истечении сессии. `AND-REQ-9` требует безопасное
хранение — пароль не сохраняется, сессионный секрет в изолированном хранилище,
запросы только по защищённому каналу. `AND-AC-4` требует модульную границу.
`AND-AC-20` подтверждает очистку секрета. `AND-AC-21` подтверждает отсутствие
сохранённого пароля. `AND-AC-22` подтверждает изоляцию сессии.

## Implementation deliverables

- Появится замкнутый класс состояния с положениями авторизации, загрузки, ошибки
  и истечения, определяющий наблюдаемый статус пользователя и управляющий
  переходами между вводом почты и вводом пароля; каждое положение содержит
  достаточно контекста для выбора шага интерфейса без дополнительной логики
  за пределами домена.
- Появятся сетевой интерфейс сервиса с методами проверки почты, входа и
  регистрации, репозиторий сессии с зашифрованным хранением секрета через
  криптографическую библиотеку платформы, и компонент локальной валидации,
  отклоняющий невалидную почту и короткий пароль до сетевого запроса.
- Появятся фокусированные модульные тесты, доказывающие переходы состояний,
  правила валидации входных данных, изоляцию сессионных данных и отсутствие
  сохранения пароля; тесты выполняются без контекста платформы и не зависят от
  компонуемого фреймворка или сети.

## Steps

Сначала написать модульные тесты для переходов состояния авторизации: начальное
положение без авторизации, переход к загрузке при проверке почты, ветвление
входа или регистрации, успех к авторизованному положению, ошибки к восстановимому
положению, истечение к положению без авторизации. Затем написать тесты валидатора
для невалидной почты, короткого пароля, пустых полей. Реализовать типизированный
контракт: замкнутый класс состояния, результат аутентификации, сетевой интерфейс
сервиса, репозиторий с зашифрованным хранилищем, валидатор. Граница приложения
остаётся чистой: действия, контекст, ресурсы и владелец жизненного цикла не входят
в держатель состояния. Граница модуля остаётся видимой через публичные типы, а
вспомогательные реализации используют внутрипакетную видимость. Обновить каталог
зависимостей с новыми версиями для сетевого клиента, парсера данных, потоков и
криптографии платформы.

## Expected result

Модуль авторизации раскрывает доменную модель аутентификации, сетевой контракт,
зашифрованное хранилище сессии и валидатор; тесты доказывают переходы положений
и правила валидации; пароль не сохраняется. Результат фиксирует проверяемый
доменный слой без привязки к интерфейсу.
