# План — авторизация профиля пользователя / Android / user-profile-auth

## Planning frame

Реализовать утверждённую продуктовую возможность авторизации как изолированный
Gradle Android library модуль `:auth`. Приложение сохраняет роль точки входа,
жизненного цикла и корневой композиции; новая возможность владеет состоянием
аутентификации, двухшаговым процессом почты и пароля, сессионным контрактом,
сетевым транспортом, защищённым хранилищем и экранами входа. Оболочка
`:app-shell` остаётся неизменной. Сеть, хранение и внедрение зависимостей в
проекте отсутствуют — они появляются внутри `:auth` через новые каталоговые
зависимости.

## Проверенные области и правила

Финальные области работы: `["application", "module", "ui"]`. Выбор подтверждён
файлами проекта: `settings.gradle.kts` содержит модули приложения и оболочки,
но не авторизации; сборка приложения включает компонуемый базовый пакет и
материальный интерфейс; оболочка представляет собой библиотеку с компонуемым
базовым пакетом; главный файл напрямую показывает оболочку без проверки сессии;
сетевые, хранилищные и внедряемые зависимости отсутствуют; общий контракт
готов и утверждён; файл пользовательского опыта готов с материальной базовой
линией. Выбранные области подтверждены текущими файлами проекта, отсутствием
слоя авторизации и утверждённым продуктовым контрактом.

Точный набор правил зафиксирован в `plan/rule-selection.json` и отражён в
`meta.json`. После этого плана реализация и проверка не добавляют новые
области работы.

## DAG

`task-001` не имеет зависимостей и создаёт физическую границу библиотеки
модуля `auth`. `task-002` зависит от `task-001` и реализует доменную модель,
данные, сетевой транспорт и защищённое хранилище. `task-003` зависит от
`task-002` и реализует компонуемые экраны авторизации с материальными
компонентами, русскими подписями и доступной семантикой. `task-004` зависит от
`task-003` и создаёт корневой маршрутизатор `AuthGate`, подключает `MainActivity`
к модулю авторизации и настраивает сетевую безопасность. `task-005` зависит от
`task-004` и выполняет интеграционную проверку: сквозные сценарии, доступность,
светлое и тёмное оформление, персистентность сессии и кросс-платформенный
паритет. Граф задач остаётся последовательным, понятным и пригодным для
поэтапной реализации без скрытых зависимостей.

## Задачи

`task-001` покрывает включение `:auth` в проект, сборку, манифест, граф
зависимостей и начальный публичный контракт. `task-002` покрывает доменную
модель аутентификации, сетевой слой, хранилище сессии, валидацию и модульные
тесты. `task-003` покрывает компонуемые экраны, локализацию, доступность,
проверки дизайн-системы, светлое и тёмное оформление и мягко-синий запасной
вариант из `platform-ux.md`. `task-004` покрывает корневую маршрутизацию,
подключение приложения к модулю авторизации и настройку сетевой безопасности.
`task-005` покрывает интеграционные тесты, доступность, проверку сессии и
паритет с утверждёнными требованиями. Каждая задача имеет собственную
ответственность и вместе они покрывают сборку, данные, интерфейс, интеграцию
и сквозную проверку.

## Estimates and multipliers

Каждая задача ограничена двумя днями. Главная неопределённость — создание
нового модуля авторизации на версии сборки с новыми сетевыми и криптографическими
зависимостями, отсутствие внедрения зависимостей в проекте и доступность эмулятора
для проверок в runtime. Если эмулятор недоступен, только runtime проверки
остаются неизвестными во время проверки, а не выводятся из статических тестов.

## Verification strategy

Для каждого наблюдаемого контракта применяется тестовое проектирование: сначала
фокусированный проваливающийся тест, затем минимальное поведение, затем узкая
команда через скрипт ограничения времени. Уже обнаруженные задачи сборки:
`app:testDebugUnitTest`, `app:lintDebug`, `app:assembleDebug`,
`app:connectedDebugAndroidTest`. Новые команды модуля остаются предварительными,
пока `task-001` не материализует модуль и поиск задач сборки не будет обновлён.

## Тестовые бюджеты

Тестовые команды используют максимум триста секунд, остановку шестьдесят секунд,
вывод двенадцатьсот строк. Команды сборки и статического анализа используют
максимум четыреста двадцать секунд, остановку девяносто секунд, вывод две
тысячи строк. Интеграция и проверки в runtime используют максимум шестьсот
секунд, остановку сто двадцать секунд, вывод две с половиной тысячи строк и
требуют обнаруженный эмулятор или устройство. Заявление о производительности
продукта в этом приращении не делается. Бюджеты конечны, чтобы проверки не
зависали и давали понятный результат.

## Контрольные точки

Перед подготовкой нужно выполнить согласование для явного набора файлов этой
пакетной версии. Сохранение допустимо только после проверки целостности
подготовленных файлов. Программный код намеренно не пишется в фазе плана.
Этот документ остаётся планом работ и не разрешает самостоятельное расширение
набора файлов.
