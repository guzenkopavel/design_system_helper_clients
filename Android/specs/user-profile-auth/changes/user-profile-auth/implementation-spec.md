# Implementation spec — user-profile-auth / Android / user-profile-auth

## Intake reference

Реализация следует утверждённому shared product contract в `specs/product/user-profile-auth/spec.md`. Платформа адаптирует наблюдаемое поведение через Android-специфичные механизмы (Material 3, защищённое хранилище, HTTP-транспорт) без изменения shared REQ/AC.

## Platform requirements

### AND-REQ-1 — Изолированная возможность оболочки *(baseline app-shell, неизменён)*

UI оболочки, состояние выбора, фиксированные подписи и нейтральные поверхности предоставлены через Android library boundary с минимальным публичным Compose-контрактом. Application entry point только компонует этот контракт и сохраняет ответственность за lifecycle и root integration.

### AND-REQ-2 — Нативное доступное представление навигации *(baseline app-shell, неизменён)*

Три утверждённых направления используют Material 3 semantics, видимые non-color признаки выбора, масштабируемый текст, русские resource labels и согласованное светлое/тёмное оформление. Совместно с `AND-REQ-10` обеспечивает кросс-платформенное паритетное поведение (`REQ-9`).

### AND-REQ-3 — Корневая композиция и гейтинг запуска

Модуль приложения (`:app`) содержит только точку входа, корневую навигацию и композицию. `MainActivity` и корневой composable определяют стартовое состояние — проверка сессии и маршрутизация между флоу авторизации и оболочкой. До получения валидной сессии оболочка и её навигация недоступны; после входа — флоу авторизации недоступен.

Covers: REQ-1

### AND-REQ-4 — Email-first флоу авторизации

Флоу авторизации предоставляется как изолированная Android-возможность через library boundary. Первый шаг — ввод адреса электронной почты; после отправки форма обращается к контракту проверки почты и по ответу переходит к шагу пароля. Шаг пароля явно показывает заголовок «Вход» или «Регистрация». Адрес почты остаётся видимым на шаге пароля; возврат к шагу почты сохраняет введённое значение.

Covers: REQ-2

### AND-REQ-5 — Переход в оболочку после успешной аутентификации

Успешный вход по существующему аккаунту или успешная регистрация новой пары почта/пароль немедленно завершают флоу авторизации и передают управление оболочке с активным разделом «Кейсы».

Covers: REQ-3

### AND-REQ-6 — Сохранение и восстановление сессии

Аутентифицированное состояние сохраняется между запусками приложения через локальное хранилище устройства на срок жизни серверной сессии. При запуске приложения контракт сессии проверяет сохранённый секрет; при валидной сессии флоу авторизации не отображается.

Covers: REQ-4

### AND-REQ-7 — Обработка ошибок флоу

Флоу авторизации отображает различимые, понятные и восстановимые состояния для каждого наблюдаемого сценария: пустые обязательные поля, синтаксически невалидная почта, пароль короче шести символов при регистрации, неверные учётные данные, занятая почта при регистрации, ограничение попыток с указанием времени повтора, отсутствие сети.

Covers: REQ-5

### AND-REQ-8 — Восстановление при истечении или отзыве сессии

Истечение или отзыв сессии при обращении к персональному запросу возвращает пользователя к шагу ввода почты с кратким объяснением. Персональные данные предыдущей сессии не отображаются. Некорректный сессионный секрет удаляется из локального хранилища.

Covers: REQ-6

### AND-REQ-9 — Безопасное хранение и передача

Пароль не сохраняется на устройстве. Сессионный секрет размещается в хранилище, недоступном другим приложениям. Учётные данные и сессионный секрет передаются только по защищённому каналу.

Covers: REQ-7

### AND-REQ-10 — Доступность и локализация

Флоу авторизации использует Material 3 компоненты и тему. Имена ассистивных элементов совпадают с видимыми русскими подписями. Ошибки, загрузка и ограничение попыток объявляются ассистивно. Смена шага переводит фокус к логическому началу нового шага. Все состояния различимы без опоры только на цвет. Поддержаны увеличенный текст, светлое, тёмное и повышенно-контрастное оформление.

Covers: REQ-8, REQ-9

## Platform acceptance criteria

### AND-AC-1 — Граница и композиция собираются *(baseline app-shell, неизменён)*

Граф Gradle включает границу библиотеки оболочки, не раскрывает детали реализации за пределами публичной Compose-границы.

Covers: AND-REQ-1

### AND-AC-2 — Состояние и UI-сценарии оболочки наблюдаемы *(baseline app-shell, неизменён)*

Focused state и Compose UI scenarios наблюдают стартовый выбор, выбор каждого направления, подписи и доступную семантику.

Covers: AND-REQ-1, AND-REQ-2

### AND-AC-3 — Нативный визуальный и accessibility fallback *(baseline app-shell, неизменён)*

Runtime evidence покрывает Material 3 light/dark, soft-blue fallback, масштабирование, TalkBack labels.

Covers: AND-REQ-2

### AND-AC-4 — Модульная граница авторизации

Gradle graph включает library-модуль авторизации с минимальным публичным контрактом; application module зависит от контракта и компонует корневую маршрутизацию, но не содержит auth domain, data или mutable state.

Covers: AND-REQ-1, AND-REQ-3

### AND-AC-5 — Корневой гейтинг по сессии

Запуск без валидной сессии открывает только флоу авторизации с полем почты; навигация оболочки не отображается. Запуск с валидной сессией немедленно открывает оболочку.

Covers: AND-REQ-3

### AND-AC-6 — Ветвление по почте: вход

Ввод почты существующего аккаунта переводит флоу на шаг пароля с заголовком «Вход».

Covers: AND-REQ-4

### AND-AC-7 — Ветвление по почте: регистрация

Ввод почты без существующего аккаунта переводит флоу на шаг пароля с заголовком «Регистрация».

Covers: AND-REQ-4

### AND-AC-8 — Возврат к шагу почты

Возврат с шага пароля к шагу почты сохраняет введённый адрес доступным для правки.

Covers: AND-REQ-4

### AND-AC-9 — Успешный вход открывает оболочку

Верные учётные данные открывают оболочку с активным разделом «Кейсы».

Covers: AND-REQ-5

### AND-AC-10 — Успешная регистрация открывает оболочку

Валидная новая пара почта/пароль создаёт аккаунт и открывает оболочку с активным разделом «Кейсы».

Covers: AND-REQ-5

### AND-AC-11 — Сессия переживает перезапуск

Перезапуск приложения в пределах жизни серверной сессии открывает оболочку без экрана входа.

Covers: AND-REQ-6

### AND-AC-12 — Неверные учётные данные

Неверный пароль показывает восстановимую ошибку без раскрытия существования аккаунта; поле пароля очищено.

Covers: AND-REQ-7

### AND-AC-13 — Пустое поле

Попытка продолжить с пустым обязательным полем не отправляет запрос и показывает человекочитаемое объяснение рядом с полем.

Covers: AND-REQ-7

### AND-AC-14 — Локальная валидация почты

Синтаксически невалидная почта отклоняется до сетевого запроса.

Covers: AND-REQ-7

### AND-AC-15 — Локальная валидация пароля

Пароль короче шести символов при регистрации отклоняется до сетевого запроса.

Covers: AND-REQ-7

### AND-AC-16 — Ограничение попыток

Ответ сервера об ограничении попыток переводит флоу в состояние с указанием времени повтора; до истечения срока основное действие не отправляет запрос.

Covers: AND-REQ-7

### AND-AC-17 — Занятая почта

Попытка регистрации на уже зарегистрированную почту показывает объяснение о существовании аккаунта с предложением выполнить вход.

Covers: AND-REQ-7

### AND-AC-18 — Офлайн-восстановление

При отсутствии сети флоу показывает отдельное восстановимое состояние с возможностью повторить; введённая почта сохраняется, пароль — нет.

Covers: AND-REQ-7

### AND-AC-19 — Истечение или отзыв сессии

Истечение или отзыв сессии возвращает на шаг ввода почты с объяснением; данные прошлой сессии не отображаются.

Covers: AND-REQ-8

### AND-AC-20 — Очистка недействительного секрета

После возврата на экран входа недействительный сессионный секрет отсутствует в хранилище.

Covers: AND-REQ-8

### AND-AC-21 — Пароль не сохранён

Проверка хранилища после входа подтверждает отсутствие сохранённого пароля.

Covers: AND-REQ-9

### AND-AC-22 — Изоляция хранилища сессии

Сессионные данные размещены в хранилище, недоступном другим приложениям.

Covers: AND-REQ-9

### AND-AC-23 — Сетевая безопасность

Все запросы авторизации передаются только по защищённому каналу; cleartext-трафик запрещён.

Covers: AND-REQ-9

### AND-AC-24 — Ассистивная семантика

Ассистивная проверка объявляет имена полей и действий, совпадающие с видимыми русскими подписями, а также ошибки, загрузку и ограничение попыток.

Covers: AND-REQ-10

### AND-AC-25 — Управление фокусом

При смене шага ассистивный фокус переходит к логическому началу нового шага.

Covers: AND-REQ-10

### AND-AC-26 — Не-цветовые индикаторы

Все состояния флоу различимы без опоры только на цвет.

Covers: AND-REQ-10

### AND-AC-27 — Масштабирование текста

При увеличенном тексте формы остаются читаемыми и работоспособными.

Covers: AND-REQ-10

### AND-AC-28 — Светлое и тёмное оформление

В светлом и тёмном оформлении подписи, сообщения и состояния сохраняют читаемость.

Covers: AND-REQ-10

### AND-AC-29 — Повышенно-контрастное оформление

В повышенно-контрастном оформлении подписи, сообщения и состояния сохраняют читаемость и различимость.

Covers: AND-REQ-10

### AND-AC-30 — Кросс-платформенный паритет

Сквозной проход флоу на Android демонстрирует идентичные шаги, тексты и правила гейтинга с iOS.

Covers: AND-REQ-2, AND-REQ-10

## Constraints

Изменение не добавляет реальный контент разделов, поведение аккаунта, пользовательский выход, сброс пароля, подтверждение почты, внешние провайдеры, продление сессии, биометрический вход, PIN-код, аналитику. `:app-shell` остаётся composition-only; auth logic не переносится в оболочку. Материал 3 — базовая линия; M3 Expressive и Dynamic color не предполагаются. Конкретные сторонние библиотеки не предписываются; спецификация описывает контракты и возможности. Адрес API — внешний параметр окружения.

## Integration points

`MainActivity.kt` — точка интеграции корневой маршрутизации: `SysDevScApp()` заменяет безусловное `AppShell` на `AuthGate`. `settings.gradle.kts` — inclusion нового модуля `:auth`. `app/build.gradle.kts` — dependency на `:auth`. `libs.versions.toml` — новые auth dependencies. Baseline `:app-shell` не меняется; `:app` зависит от `:auth` и `:app-shell`, компонует `AuthGate` как корневой маршрутизатор.
