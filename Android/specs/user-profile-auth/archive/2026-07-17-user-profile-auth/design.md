# Design — user-profile-auth / Android / user-profile-auth

## Current context

Текущий граф Gradle содержит два модуля: модуль приложения с точкой входа и модуль-библиотеку с оболочкой для трёх направлений. Точка входа напрямую отрисовывает оболочку без проверки сессии, без флоу авторизации, без сетевого слоя, без защищённого хранилища и без фреймворка внедрения зависимостей. Существующие границы авторизации, навигации, данных, сети или хранения, пригодные для переиспользования, отсутствуют. Доставленная оболочка зафиксирована платформенной спецификацией.

## Proposed architecture and boundaries

Предполагаемый модуль-библиотека в графе Gradle владеет цельной возможностью авторизации: состоянием аутентификации, двухшаговым флоу (почта и пароль), контрактом сессии, сетевым транспортом, защищённым хранилищем и экранами входа. Модуль приложения сохраняет жизненный цикл точки входа, корневую композицию, конфигурацию и ресурсы; он зависит от модуля авторизации и компонует корневой маршрутизатор. Оболочка остаётся неизменной.

## Modularity decision

- Outcome: isolated
- Capability triggers: independent-feature=yes; domain-data=yes; network=yes; persistence=yes; reusable-ui=no; consumers=1; independent-ownership=yes
- Physical boundaries: Proposed Gradle Android library module for auth capability isolation
- Public contracts and dependency direction: Публичная граница принимает обратный вызов при успешной проверке и показывает флоу авторизации, когда пользователь не подтверждён; замкнутый класс описывает состояние сессии; код потребителя зависит от этого контракта.
- App-shell responsibilities: entry-points, lifecycle, root-routing, dependency-wiring, platform-configuration, target-resources
- App-shell capability ownership: none
- Repository evidence: Android/settings.gradle.kts; Android/build.gradle.kts; Android/gradle/libs.versions.toml
- Rationale and trade-offs: Возможность авторизации с двухшаговым флоу, состоянием сессии, сетевым транспортом и защищённым хранилищем образует цельную самостоятельно владеемую единицу; выделение защищает корневую композицию и даёт детерминированные тесты состояния.
- Migration boundary and trigger: Миграция не требуется, потому что proposed physical boundary появляется до implementation; будущие дополнительные consumers сохраняют тот же public seam.
- Over-modularization check: Одна цельная возможность авторизации держит экраны, состояние сессии, валидацию и тесты вместе; сетевой транспорт и хранение сессии изолируются внутри того же модуля; лишняя нагрузка на сборку не создаётся.
- Boundary guard verdict: PASS

## Data and control flow

При запуске `MainActivity` → `SysDevScApp()` → `AuthGate`. `AuthGate` проверяет сохранённый сессионный секрет: при валидной сессии немедленно показывает `AppShell`; при невалидной — `AuthScreen` с шагом почты. Пользователь вводит почту → `email-check` → `AuthScreen` показывает шаг пароля с заголовком «Вход» или «Регистрация». Отправка пароля → `login`/`register` → при успехе сохраняет сессию и показывает `AppShell`; при ошибке — показывает состояние в контексте шага. 401 на персональный запрос после входа → возврат на шаг почты с очисткой данных. Русское уточнение: данные потока — почта, пароль, сессионный секрет; контроль — состояние сессии и переходы между шагами.

## Error and recovery model

Загрузка: основное действие заблокировано с inline `CircularProgressIndicator` во время `email-check`, `login`, `register` и проверки сессии при запуске. Пустое поле: поддерживающий текст без отправки запроса. Локальная валидация: невалидная почта и пароль короче шести символов отклоняются до сетевого запроса. Серверные ошибки: неверные учётные данные, занятая почта — сообщения без раскрытия существования аккаунта. Rate limit: 429 ответ переводит floau в состояние с указанием времени повтора; повторная отправка заблокирована. Offline: отдельное восстановимое состояние с сохранением почты. Session expiry: 401 при персональном запросе возвращает на шаг почты с очисткой данных и сессионного секрета. Русское уточнение: каждое состояние восстановимо, и пользователь не застревает.

## Platform UX trace and decisions

`platform-ux.md` имеет статус готовности и выбирает Material 3 через обнаруженную зависимость. Семантические мягко-синие роли основного действия и фокуса, тихие нейтральные поверхности, русские подписи из ресурсов, текстовые признаки ошибок и загрузки, масштабируемая типографика и запасные варианты оформления направляют публичную границу. Выразительное оформление и динамические цвета исключены до подтверждения наличия в репозитории.

## Verification strategy

План создаёт детерминированные тесты состояния для переходов аутентификации, сценарии экранов для смены шагов и состояний ошибок, проверки модульной границы, а также фокусированные доказательства доступного представления. Интеграция потребителя собирает корневую композицию через маршрутизатор; проверка хранилища подтверждает изоляцию сессионных данных и отсутствие сохранённого пароля. Проверка конфигурации сетевой безопасности подтверждает запрет незащищённого трафика.

## Applied engineering scopes

- application: Существующая точка входа сохраняет только композицию и корневую маршрутизацию; логика авторизации и изменяемое состояние находятся в модуле-библиотеке.
- module: Независимая возможность авторизации изолируется в новом модуле-библиотеке Gradle с минимальной публичной гранью.
- ui: Material 3 компоненты (`OutlinedTextField`, `FilledTonalButton`, `CircularProgressIndicator`), семантические роли, accessibility semantics, font scale, appearance и non-color cues требуют focused presentation verification, отдельной проверки доступности и понятного поведения для пользователя.
- Русское пояснение: выбранные области фиксируют ответственность, сборку и визуальную доступность без добавления DI framework, multiplatform или extra data/network layers.
- Дополнительное русское пояснение: список областей нужен для реализации утверждённой authorization capability, а не для расширения архитектуры за пределы текущего продуктового решения.

## Constraints

Изменение не добавляет реальный контент разделов, поведение аккаунта, пользовательский выход, сброс пароля, подтверждение почты, внешние провайдеры, продление сессии, биометрический вход, PIN-код, аналитику. `:app-shell` остаётся composition-only; auth logic не переносится в оболочку. Материал 3 — базовая линия; M3 Expressive и Dynamic color не предполагаются. Конкретные сторонние библиотеки (HTTP-клиент, JSON-парсер, корутины, DI) не предписываются; спецификация описывает контракты и возможности. Адрес API — внешний параметр окружения.

## Integration points

`MainActivity.kt` — точка интеграции корневой маршрутизации: `SysDevScApp()` заменяет безусловное `AppShell` на `AuthGate`. `settings.gradle.kts` — inclusion нового модуля `:auth`. `app/build.gradle.kts` — dependency на `:auth`. `libs.versions.toml` — новые auth dependencies. Baseline `:app-shell` не меняется; `:app` зависит от `:auth` и `:app-shell`, компонует `AuthGate` как корневой маршрутизатор.
