# Proposal — user-profile-auth / Android / user-profile-auth

## Intake

Это product-backed изменение для Android. `specs/product/user-profile-auth/spec.md` имеет статус `READY` и явное `APPROVED`; проверенный общий контракт остаётся единственным источником наблюдаемого поведения, шагов, текстов и правил гейтинга. Общий `review-verdicts.json` подтверждает полноту шести линз ревью.

## Goal

Добавить email-first флоу авторизации в Android-клиент: приложение гатится проверкой сессии, без валидной сессии пользователь видит только флоу входа/регистрации, после успешной аутентификации открывается доставленная оболочка `app-shell`. Сессия сохраняется между запусками на срок её жизни.

## Scope

Изменение вводит proposed Gradle Android library module для auth capability, Compose-экраны авторизации (два шага: почта и пароль), корневой маршрутизатор `AuthGate`, сохранение сессии через защищённое хранилище, HTTP-транспорт для email-check/login/register/profile, обработку наблюдаемых состояний (загрузка, валидация, ошибки, ограничение попыток, офлайн, истечение сессии) и русские строковые ресурсы. Документ ссылается на shared `REQ-1`–`REQ-9` и `AC-1`–`AC-25`, но не копирует их текст и не ослабляет общий контракт.

## Engineering scope selection

`application` покрывает корневую маршрутизацию и composition в `MainActivity`; `module` нужен для изоляции auth capability в новый Gradle Android library module с минимальным публичным контрактом; `ui` подтверждает Compose Material 3 для экранов авторизации. Для `compose`, `gradle`, `multiplatform`, `localization`, `delivery` и `developer-experience` нет repository evidence: `compose` не является отдельной инженерной областью в этом пакете (UI покрывает Compose), `gradle` не выбирается, так как wiring модуля покрывается `module`, `multiplatform` отсутствует, `localization` не является отдельной областью (тексты входят в `ui`), `delivery` и `developer-experience` не применимы.

## Applicable rule files

Точный набор правил lifecycle для Propose зафиксирован в `meta.json`: общие правила спецификации, системного дизайна и модульности; Android-архитектура и modularization; а также профили выбранных областей инженерной работы (application, module, ui).

## Non-goals

Изменение не добавляет пользовательский выход («Выйти»), наполнение раздела «Профиль», сброс пароля, подтверждение почты, внешние провайдеры, продление сессии, биометрический вход, PIN-код, офлайн-кэш контента, реальное содержимое разделов оболочки, аналитику, изменение веба или бэкенда. Реализация для iOS остаётся вне этого Android package.

## Accepted decisions

Независимым владельцем возможности авторизации выбран новый Gradle-модуль библиотеки; существующая точка входа приложения сохраняет только ответственность за композицию и маршрутизацию. Material 3 выступает нативной базой оформления; `platform-ux.md` переводит общий спокойный мягко-синий намерение в семантические роли и доступные запасные решения. Сессионный секрет хранится через защищённый платформенный механизм.

## Open questions

None.

## Risks

Защищённое хранилище сессии требует выбора конкретного платформенного механизма (AndroidX Security Crypto или DataStore) и introduction DI framework, если он не выбран для manual DI. Сетевой транспорт — greenfield; нужен HTTP-клиент с поддержкой cookie-jar и TLS. Доступность Android runtime, масштабирование шрифта, семантика TalkBack, тёмное оформление и high-contrast presentation требуют focused evidence, а не предположений.
