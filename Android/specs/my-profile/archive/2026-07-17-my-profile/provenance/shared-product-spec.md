## Problem and Why

Авторизованный пользователь уже попадает в оболочку приложения, но вкладка
профиля не даёт базового контроля над персональным контекстом. Пользователь не
видит, под какой почтой он вошёл, не может быстро узнать, есть ли у него
собственные интервью, и не имеет явного выхода из сессии. Для доверия к
персональным функциям и смены аккаунта нужен простой профильный экран,
одинаковый для iOS и Android.

## Outcomes and Success Signals

- Пользователь открывает вкладку профиля и сразу видит email текущего аккаунта.
- Пользователь понимает, доступна ли его история интервью, без перехода в
  отдельный экран.
- При наличии интервью одно нажатие показывает количество интервью.
- При отсутствии интервью действие «Мои интервью» не допускает ошибочного
  нажатия.
- Выход завершает сессию и возвращает пользователя на чистый экран ввода почты.

## Scope

- Авторизованная вкладка профиля в существующей оболочке из трёх вкладок.
- Крупная стандартная профильная иконка, email текущего пользователя и блок
  действий «Мои интервью» / «Выход».
- Загрузка текущего профиля и всех страниц истории интервью текущего профиля
  для определения полного количества.
- Условное доступное или недоступное поведение действия «Мои интервью».
- Краткое сообщение с количеством интервью.
- Явный выход и возврат на первый экран auth-флоу с пустой почтой.
- Состояния загрузки, пустой истории, ошибки, отсутствия сети,
  недействительной сессии, доступности, локализации, приватности и паритета
  для iOS/Android.

## Non-Goals

- Экран списка интервью, детали интервью, продолжение интервью, фильтры истории
  или навигация в интервью.
- Отображение содержимого интервью, названий кейсов, оценок, прогресса или
  дат на профиле.
- Редактирование профиля, пользовательский аватар, имя, настройки аккаунта,
  удаление аккаунта или смена пароля.
- Изменение auth API, interview history API, backend, веб-клиента или других
  двух вкладок оболочки.
- Новая аналитика.
- Платформенная архитектура, SDK/framework/component выбор, конкретные классы,
  модули или способ хранения сессии.

## Product Decisions

- `my-profile` является отдельной продуктовой фичей, а
  `user-profile-auth` остаётся зависимым baseline для авторизованного состояния
  и восстановления при недействительной сессии.
- Профиль показывает только данные, подтверждённые текущим API: email текущего
  профиля. Технический `id` не отображается.
- Профильный символ стандартный и крупный, расположен по центру с небольшим
  верхним отступом. Он не является пользовательским аватаром и не редактируется.
- «Мои интервью» всегда видима в блоке действий, но доступна только при полном
  известном количестве интервью больше нуля.
- Полное количество интервью определяется догрузкой страниц
  `GET /api/interviews/history` до завершения пагинации; прямое поле `total`
  не требуется.
- Нажатие доступной «Мои интервью» показывает краткое сообщение
  «Интервью: N» и не открывает новый экран.
- «Выход» доступен независимо от состояния загрузки истории; успешное завершение сессии
  возвращает на экран ввода почты с пустым полем.
- Ошибка истории не блокирует выход; недействительная сессия возвращает к
  базовому восстановлению auth-флоу без персональных данных.

## Shared Observable Behavior

### Happy path

1. Авторизованный пользователь выбирает вкладку профиля.
2. Приложение загружает текущий профиль и историю интервью текущей серверной
   сессии, догружая страницы истории до завершения пагинации.
3. Экран профиля показывает крупную стандартную профильную иконку по центру,
   ниже email текущего пользователя, ниже блок с действиями «Мои интервью» и
   «Выход».
4. Если полное количество интервью `N > 0`, «Мои интервью» доступна.
5. Нажатие «Мои интервью» показывает краткое сообщение «Интервью: N» и
   оставляет пользователя на вкладке профиля.
6. Нажатие «Выход» завершает сессию и открывает экран ввода почты; поле почты
   пустое.

### Secondary states

- **Загрузка:** до завершения загрузки профиля и полного количества интервью
  экран не показывает ложные значения. «Мои интервью» не активируется, пока
  счётчик неизвестен. Во время выхода повторная отправка невозможна.
- **Пустая история:** если полное количество интервью равно `0`, «Мои интервью» видима,
  но недоступна для нажатия.
- **Ошибка:** ошибка загрузки истории объясняется восстановимо; «Мои интервью»
  остаётся недоступной до успешного определения счётчика; «Выход» остаётся доступным.
- **Нет сети:** при отсутствии сети профиль показывает восстановимое состояние
  для истории. Если выход не может завершиться из-за сети, пользователь
  остаётся в профиле с сообщением и без ложного состояния выхода.
- **Недействительная сессия:** ответ о недействительной сессии при загрузке профиля или
  истории возвращает пользователя в auth-флоу по базовому контракту; email и данные
  истории прошлого профиля не отображаются.
- **Доступность:** email, действия, недоступное состояние, краткое сообщение и
  занятость выхода объявляются ассистивно; состояния различимы не только цветом.
- **Локализация:** видимые пользовательские тексты фиксированы на русском:
  «Мои интервью», «Выход», «Интервью: N».
- **Аналитика и приватность:** новая аналитика не добавляется; история используется
  только для счётчика и доступного или недоступного состояния; детали интервью на профиле не
  отображаются.

## Requirements

- `REQ-1` — Авторизованная вкладка профиля отображает текущий аккаунт через
  крупную стандартную профильную иконку, email пользователя и блок действий
  «Мои интервью» / «Выход».
- `REQ-2` — Приложение определяет полное количество интервью текущего профиля
  по истории интервью текущей серверной сессии.
- `REQ-3` — Действие «Мои интервью» доступно только при полном известном
  количестве интервью больше нуля.
- `REQ-4` — Нажатие доступного действия «Мои интервью» показывает краткое
  сообщение с количеством интервью текущего профиля и не меняет экран.
- `REQ-5` — При отсутствии интервью действие «Мои интервью» остаётся видимым,
  но недоступным для активации.
- `REQ-6` — Действие «Выход» завершает текущую сессию и возвращает пользователя
  на экран ввода почты с пустым полем почты.
- `REQ-7` — Ошибки загрузки истории, отсутствие сети и недействительная сессия
  отображаются как восстановимые или базовые recovery-состояния без раскрытия
  персональных данных после потери авторизации.
- `REQ-8` — Поведение профиля, доступности действий, toast-сообщения и выхода
  одинаково для iOS и Android на уровне наблюдаемого продукта.
- `REQ-9` — Экран профиля доступен ассистивно: email, действия,
  недоступное состояние, сообщение и занятость выхода имеют корректную семантику.
- `REQ-10` — Экран профиля поддерживает увеличенный текст, светлое, тёмное и
  повышенно-контрастное оформление, а состояния различимы не только цветом.

## Acceptance Criteria

- `AC-1` — При открытии профиля авторизованный пользователь видит крупную профильную иконку по центру, свою почту и блок с действиями «Мои интервью» и «Выход». `Covers: REQ-1` `Verification dimension: profile-layout-content`
- `AC-2` — Для постраничной истории приложение продолжает загрузку страниц до завершения пагинации перед тем, как считать количество интервью полным. `Covers: REQ-2` `Verification dimension: interview-history-pagination`
- `AC-3` — Если полное количество интервью текущего профиля больше нуля, «Мои интервью» доступна для активации. `Covers: REQ-3` `Verification dimension: my-interviews-enabled`
- `AC-4` — Нажатие доступной «Мои интервью» показывает краткое сообщение «Интервью: N», где `N` равно полному количеству интервью текущего профиля. `Covers: REQ-4` `Verification dimension: my-interviews-count-toast`
- `AC-5` — После показа сообщения с количеством пользователь остаётся на вкладке профиля. `Covers: REQ-4` `Verification dimension: my-interviews-no-navigation`
- `AC-6` — Если полное количество интервью текущего профиля равно нулю, «Мои интервью» видима, но недоступна для нажатия. `Covers: REQ-5` `Verification dimension: my-interviews-empty-disabled`
- `AC-7` — Нажатие «Выход» при успешном logout возвращает на экран ввода почты, и поле почты пустое. `Covers: REQ-6` `Verification dimension: logout-empty-email-entry`
- `AC-8` — После успешного выхода email и данные истории прошлого профиля не отображаются в оболочке. `Covers: REQ-6` `Verification dimension: logout-clears-profile-display`
- `AC-9` — При ошибке загрузки истории или offline-состоянии «Выход» остаётся доступным, а «Мои интервью» не активируется до успешного определения полного количества. `Covers: REQ-7` `Verification dimension: history-error-action-availability`
- `AC-10` — Ответ о недействительной сессии при загрузке профиля или истории возвращает пользователя в auth-флоу по базовому контракту без отображения персональных данных прошлой сессии. `Covers: REQ-7` `Verification dimension: invalid-session-profile-recovery`
- `AC-11` — Сетевой сбой logout оставляет пользователя в профиле с восстановимым сообщением и без ложного перехода на экран входа. `Covers: REQ-7` `Verification dimension: logout-failure-recovery`
- `AC-12` — Один и тот же набор состояний профиля, правил доступности, текста сообщения и результата выхода наблюдается на iOS и Android. `Covers: REQ-8` `Verification dimension: cross-client-profile-parity`
- `AC-13` — Ассистивная проверка объявляет email текущего профиля и имена действий «Мои интервью» / «Выход» согласно видимым подписям. `Covers: REQ-9` `Verification dimension: assistive-profile-semantics`
- `AC-14` — Ассистивная проверка объявляет недоступность «Мои интервью», сообщение «Интервью: N» и состояние занятости выхода. `Covers: REQ-9` `Verification dimension: assistive-feedback-states`
- `AC-15` — При увеличенном тексте профильная иконка, email и блок действий остаются читаемыми и не перекрываются. `Covers: REQ-10` `Verification dimension: profile-text-scaling`
- `AC-16` — В светлом и тёмном оформлении email, действия, недоступное состояние и сообщение сохраняют читаемость. `Covers: REQ-10` `Verification dimension: profile-light-dark-appearance`
- `AC-17` — В повышенно-контрастном оформлении email, действия, недоступное состояние и сообщение сохраняют читаемость и различимость. `Covers: REQ-10` `Verification dimension: profile-increased-contrast`
- `AC-18` — Доступное, недоступное, загрузочное, ошибочное и критическое состояние выхода различимы без опоры только на цвет. `Covers: REQ-10` `Verification dimension: profile-non-color-cues`

## Common Product Constraints

- `my-profile` требует валидного авторизованного состояния из baseline
  `user-profile-auth`.
- API и backend-контракт не меняются этим изменением.
- Текущий профиль берётся из `GET /api/profile`; отображается только `email`.
- История интервью берётся из `GET /api/interviews/history` для текущей
  серверной сессии; endpoint защищён `SessionCookie`.
- Прямого поля общего количества интервью в обнаруженной схеме нет; полный
  счётчик определяется догрузкой страниц истории до `page.hasMore = false`.
- Выход использует `POST /api/auth/logout`, который очищает `SessionCookie` и
  возвращает `204` для active, missing или invalid sessions.
- Адрес API остаётся внешним параметром окружения, не продуктовым решением.
- Учётные и сессионные данные продолжают передаваться только по защищённому
  каналу согласно baseline auth-контракту.
