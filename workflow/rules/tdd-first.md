# Behavior-first и TDD

Для нового или изменённого наблюдаемого поведения сначала получить RED,
доказывающий отсутствие нужного результата, затем внести минимальный код для
GREEN и только после этого выполнять REFACTOR.

## Цикл

1. Выбрать один контракт и минимальный наблюдаемый пример.
2. Запустить тест и убедиться, что он падает по ожидаемой причине, а не из-за
   окружения или синтаксиса.
3. Реализовать минимальное поведение без побочных «улучшений».
4. Повторить focused test и релевантный regression set.
5. Рефакторить только при сохранённом green.
6. Сохранить команды и результат как task/verification evidence.

Тест, который сразу зелёный, не доказывает новый контракт. Его нужно исправить
или зафиксировать, что задача является characterization/refactor и поведение уже
существует.

## Допустимые исключения

Generated code, чистые документы, механические metadata edits и исследовательский
spike могут не начинаться с executable test. Исключение записывается в task с
альтернативным RED: lint failure, missing artifact, reproducible diagnostic или
manual observation. Spike-код не становится production-кодом без обычного
цикла.

Bug fix начинается с воспроизводящего regression test на самом низком
достаточном уровне. Mock-heavy тест не заменяет проверку реальной интеграционной
границы, если дефект находился на ней.
