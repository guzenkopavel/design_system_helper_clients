# Code comments

Комментарий — исключение. Сначала улучшить имя, тип или структуру. Комментарий
нужен, только если без него инженер сделает неверный вывод о причине решения.

## Допустимо

- объяснить скрытый invariant, ordering или lifecycle constraint;
- зафиксировать workaround со ссылкой на проверяемую внешнюю проблему;
- объяснить выбор измеренного лимита или threshold с единицами;
- описать security/privacy rationale, не видимый из сигнатуры;
- дать doc-comment публичному контракту, описывая обязательства и ошибки.

## Запрещено

- пересказывать строку кода, имя функции или историю изменения;
- оставлять закомментированный код;
- писать self-praise, сведения об авторе или генераторе;
- ссылаться из production-кода на временные task/change identifiers;
- оставлять `TODO`/`FIXME` без trackable reference и условия удаления;
- утверждать версию SDK, поведение framework или performance-число без
  проверяемого основания.

Комментарий должен оставаться верным после переименования соседнего типа. Если
контекст принадлежит решению, плану или delivery history, он хранится в
соответствующем артефакте, а не в production-файле.

Перед завершением изменения поискать stale TODO, commented-out code и
комментарии «что делает код». Формат и язык комментариев уточняет платформа.
